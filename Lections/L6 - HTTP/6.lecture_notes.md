# Лекція 6. Протокол HTTP

## 1. Від мережевого з’єднання до прикладного протоколу

Для обміну даними двом програмам недостатньо лише фізичного або мережевого з’єднання. Вони повинні домовитися, як саме інтерпретувати передані байти.

**Мережевий протокол** — це набір правил, який визначає:

- формат повідомлень;
- порядок їх передавання;
- призначення окремих полів;
- очікувану поведінку сторін;
- способи повідомлення про помилки.

На транспортному рівні вебзастосунки традиційно використовують TCP. Операційна система надає програмі сокет, через який можна читати й записувати байти.

Роботу із сокетом можна приблизно порівняти з роботою з файлом:

```text
файл   → байти записуються на диск і читаються з диска;
сокет  → байти передаються іншій програмі через мережу.
```

TCP забезпечує надійний двосторонній потік байтів, але не пояснює зміст цих байтів. Він не визначає, де починається документ, який його тип, скільки він має байтів і чи успішно виконано запит.

Ці правила задає протокол прикладного рівня — HTTP.

## 2. Що таке HTTP

HTTP розшифровується як **Hypertext Transfer Protocol** — протокол передавання гіпертексту.

Спочатку він створювався для отримання гіпертекстових документів, але згодом став універсальним протоколом взаємодії між програмами.

Через HTTP передають:

- HTML;
- CSS;
- JavaScript;
- JSON;
- XML;
- зображення;
- аудіо й відео;
- файли;
- команди й результати їх виконання;
- дані між мікросервісами.

HTTP не прив’язаний до браузера. Клієнтом може бути:

- браузер;
- мобільний застосунок;
- консольна утиліта;
- настільна програма;
- інший сервер;
- автоматизований агент;
- пристрій Інтернету речей.

Перевага використання HTTP полягає в тому, що для нього існують готові сервери, клієнтські бібліотеки, проксі, засоби захисту та інструменти налагодження.

## 3. HTTP поверх TCP

У спрощеному вигляді взаємодія виглядає так:

```text
клієнт
  ↓ встановлення TCP-з’єднання
сервер
  ↓ HTTP-запит
сервер обробляє запит
  ↓ HTTP-відповідь
клієнт
```

TCP надає канал передавання, а HTTP визначає структуру інформації в цьому каналі.

Для HTTPS перед обміном HTTP-повідомленнями додатково виконується TLS-рукостискання.

## 4. Модель «запит — відповідь»

Класична модель HTTP:

```text
request → response
```

Клієнт створює запит, а сервер повертає одну відповідь.

Наприклад:

```text
браузер: надай /index.html
сервер: 200 OK + документ
```

Наступний запит є окремою операцією:

```text
браузер: надай /style.css
сервер: 200 OK + таблиця стилів
```

Навіть для однієї сторінки браузер зазвичай надсилає багато запитів: окремо для HTML, CSS, JavaScript, шрифтів і зображень.

## 5. Відсутність стану

HTTP називають протоколом без стану, або **stateless protocol**.

Це означає, що кожен запит повинен містити інформацію, достатню для його обробки. Сам протокол не вимагає, щоб сервер автоматично пам’ятав попередні запити того самого клієнта.

Наприклад, сервер отримує:

```text
GET /profile
```

Із самого запиту без додаткових даних не видно:

- хто саме звернувся;
- чи входив користувач до системи;
- які сторінки він відкривав раніше;
- що знаходиться в його кошику.

Для збереження логічного стану використовуються додаткові механізми:

- cookie;
- сесії;
- токени;
- параметри URL;
- приховані поля;
- дані в тілі запиту.

## 6. Формат HTTP-повідомлення

HTTP/1.x використовує текстове представлення стартового рядка й заголовків.

Повідомлення складається з:

1. стартового рядка;
2. заголовків;
3. порожнього рядка;
4. необов’язкового тіла.

```text
стартовий рядок
Header-Name: value
Header-Name: value

тіло
```

Порожній рядок відокремлює заголовки від тіла.

## 7. Структура HTTP-запиту

Приклад:

```http
GET /courses/web HTTP/1.1
Host: example.com
Accept: text/html
Accept-Language: uk
User-Agent: ExampleBrowser/1.0
```

### 7.1. Стартовий рядок

```http
GET /courses/web HTTP/1.1
```

Він містить:

- `GET` — метод;
- `/courses/web` — ціль запиту;
- `HTTP/1.1` — версію протоколу.

### 7.2. Заголовки

```http
Host: example.com
Accept: text/html
```

Заголовки передають метаінформацію: домен, типи даних, мову, кодування, авторизацію, cookie та інші параметри.

### 7.3. Тіло

Для `GET` тіло зазвичай відсутнє. У `POST`, `PUT` або `PATCH` воно часто містить дані.

```http
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 36

{"name":"Anna","group":"KN-101"}
```

## 8. Заголовок `Host`

Одна IP-адреса може обслуговувати багато доменів. Тому серверу потрібно знати, для якого сайта призначено запит.

```http
Host: example.com
```

У HTTP/1.1 заголовок `Host` є обов’язковим.

Без нього сервер не завжди може вибрати потрібний віртуальний хост.

## 9. Структура HTTP-відповіді

Приклад:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 44

<h1>Основи вебтехнологій</h1>
```

### 9.1. Рядок статусу

```http
HTTP/1.1 200 OK
```

Він містить:

- версію протоколу;
- числовий код стану;
- текстове пояснення.

### 9.2. Заголовки

Вони описують сервер і тіло відповіді.

### 9.3. Тіло

Тіло може містити HTML, JSON, зображення або інший ресурс.

Для деяких відповідей тіло відсутнє, наприклад для `204 No Content` або відповіді на `HEAD`.

## 10. MIME-типи

Заголовок `Content-Type` повідомляє, як інтерпретувати тіло.

```http
Content-Type: text/html; charset=UTF-8
```

Поширені типи:

```text
text/plain
text/html
text/css
application/javascript
application/json
image/png
image/jpeg
application/pdf
application/octet-stream
```

Однакові байти можуть бути оброблені по-різному залежно від MIME-типу.

Нехай сервер повертає:

```html
<b>Вітаємо</b>
```

Якщо заголовок:

```http
Content-Type: text/plain
```

браузер покаже теги як текст.

Якщо:

```http
Content-Type: text/html
```

браузер розбере HTML і покаже жирний текст.

Отже, розширення файла саме по собі не визначає спосіб обробки HTTP-відповіді. Вирішальним є заголовок `Content-Type`.

## 11. Довжина та межі тіла

TCP є потоком байтів і не позначає межі HTTP-документа. Тому клієнт повинен розуміти, скільки байтів належить до відповіді.

Один із варіантів:

```http
Content-Length: 1024
```

Клієнт очікує рівно 1024 байти тіла.

Інші способи залежать від версії HTTP та режиму передавання. Наприклад, у HTTP/1.1 може використовуватися chunked transfer encoding.

## 12. HTTP-методи

Метод описує бажану операцію над ресурсом.

### 12.1. `GET`

Отримання представлення ресурсу:

```http
GET /articles/10 HTTP/1.1
```

`GET` не повинен змінювати стан сервера як основний результат виконання.

### 12.2. `POST`

Надсилання даних або створення нового ресурсу:

```http
POST /articles HTTP/1.1
Content-Type: application/json

{"title":"HTTP"}
```

### 12.3. `HEAD`

Подібний до `GET`, але сервер повертає лише заголовки.

Використання:

- перевірка існування ресурсу;
- отримання довжини;
- перевірка часу зміни;
- перевірка кешу.

### 12.4. `PUT`

Повна заміна ресурсу за відомою адресою.

### 12.5. `PATCH`

Часткова зміна ресурсу.

### 12.6. `DELETE`

Видалення ресурсу.

Важливо: метод є частиною домовленості. HTTP не змушує програму автоматично виконати відповідну операцію. Конкретну поведінку реалізує серверний код.

## 13. Безпечність та ідемпотентність

**Безпечний метод** не повинен змінювати стан ресурсу. Приклад — `GET`.

**Ідемпотентний метод** дає той самий кінцевий результат після одного або кількох однакових викликів.

Зазвичай ідемпотентними вважають:

- `GET`;
- `HEAD`;
- `PUT`;
- `DELETE`.

`POST` зазвичай не є ідемпотентним: повторне надсилання може створити два ресурси.

## 14. Передавання параметрів у URL

Параметри після `?` називаються query string:

```text
/search?q=http&page=2
```

Тут:

- `q=http`;
- `page=2`.

Query string зручний для:

- пошуку;
- фільтрації;
- сортування;
- пагінації;
- стану, який можна зберегти в посиланні.

Для великих або чутливих даних краще використовувати тіло запиту.

## 15. Передавання форм

Звичайна HTML-форма може передавати дані як:

```http
Content-Type: application/x-www-form-urlencoded
```

Тіло:

```text
name=Anna&group=KN-101
```

Для файлів застосовується:

```http
Content-Type: multipart/form-data; boundary=...
```

Тіло поділяється на окремі частини, кожна з власними заголовками.

## 16. Бінарні дані та Base64

HTTP може передавати бінарні дані без перетворення, якщо межі й тип вмісту визначено коректно.

Base64 перетворює байти на текстові символи. Це буває зручно, коли канал або формат очікує текст, але має недоліки:

- розмір збільшується приблизно на третину;
- потрібне кодування та декодування;
- це не шифрування;
- дані залишаються доступними для відновлення.

Для завантаження файлів у формах зазвичай доцільніше використовувати `multipart/form-data`.

## 17. Основні заголовки запиту

### `User-Agent`

Ідентифікує програму-клієнта.

```http
User-Agent: Mozilla/5.0 ...
```

На нього не слід покладатися як на надійний доказ, оскільки клієнт може встановити довільне значення.

### `Accept`

Повідомляє, які MIME-типи клієнт готовий отримати.

```http
Accept: text/html, application/json
```

### `Accept-Language`

Бажані мови:

```http
Accept-Language: uk, en;q=0.8
```

### `Accept-Encoding`

Підтримувані способи стиснення:

```http
Accept-Encoding: gzip, br
```

### `Cookie`

Передає збережені cookie:

```http
Cookie: session_id=abc123
```

## 18. Основні заголовки відповіді

### `Content-Type`

Тип тіла.

### `Content-Length`

Кількість байтів.

### `Content-Encoding`

Спосіб стиснення:

```http
Content-Encoding: gzip
```

### `Location`

Нова адреса під час перенаправлення:

```http
Location: https://example.com/new-address
```

### `Set-Cookie`

Команда браузеру зберегти cookie:

```http
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax
```

## 19. Коди стану

Перша цифра визначає клас відповіді.

### 19.1. `1xx` — інформаційні

Проміжні повідомлення про перебіг взаємодії.

### 19.2. `2xx` — успішні

Запит прийнято й успішно оброблено.

### 19.3. `3xx` — перенаправлення

Для завершення операції клієнту потрібно використати іншу адресу або локальний кеш.

### 19.4. `4xx` — помилка клієнта

Запит некоректний або клієнт не має необхідних прав.

### 19.5. `5xx` — помилка сервера

Запит міг бути коректним, але сервер не зміг його обробити.

## 20. Типові коди

### `200 OK`

Успішна відповідь із вмістом.

### `201 Created`

Новий ресурс створено.

### `204 No Content`

Операцію виконано, але тіло не повертається.

`204` не означає «використай кеш». Для підтвердження, що кешована копія не змінилася, використовується `304`.

### `301 Moved Permanently`

Ресурс постійно перенесено.

### `302 Found`

Тимчасове перенаправлення в поширеному браузерному використанні.

### `304 Not Modified`

Ресурс не змінився. Клієнт може використати локальну копію.

### `400 Bad Request`

Сервер не може розібрати або прийняти запит.

### `401 Unauthorized`

Для доступу потрібна автентифікація або надані облікові дані неприйнятні.

Попри назву, за змістом цей код пов’язаний насамперед з автентифікацією.

### `403 Forbidden`

Сервер розпізнав запит, але забороняє доступ.

### `404 Not Found`

Ресурс не знайдено.

### `500 Internal Server Error`

Непередбачена помилка в серверному застосунку.

### `502 Bad Gateway`

Проксі або шлюз не отримав коректної відповіді від наступного сервера.

### `503 Service Unavailable`

Сервіс тимчасово недоступний, наприклад через перевантаження або технічні роботи.

## 21. Cookie

HTTP не пам’ятає користувача між запитами, тому сервер може попросити браузер зберегти невелике значення:

```http
Set-Cookie: session_id=7f8a...; Path=/; HttpOnly
```

Браузер зберігає cookie для відповідного домену й автоматично додає його до наступних запитів:

```http
Cookie: session_id=7f8a...
```

Cookie може зберігати:

- ідентифікатор сесії;
- налаштування;
- мовні параметри;
- технічні маркери.

Не слід зберігати в cookie відкритий пароль.

## 22. Серверна сесія

Типова схема:

1. користувач надсилає логін і пароль;
2. сервер перевіряє їх;
3. сервер створює запис сесії;
4. генерується складний випадковий ідентифікатор;
5. сервер відправляє його через `Set-Cookie`;
6. браузер додає cookie до наступних запитів;
7. сервер знаходить сесію за ідентифікатором.

```text
session_id → запис у пам’яті або базі даних
```

Якщо видалити cookie, браузер перестане надсилати ідентифікатор і користувач втратить авторизовану сесію.

Якщо зловмисник викраде чинний ідентифікатор, він може спробувати видавати себе за користувача. Тому важливі:

- HTTPS;
- випадковий ідентифікатор достатньої довжини;
- `HttpOnly`;
- `Secure`;
- `SameSite`;
- обмежений час життя;
- перевидача ідентифікатора після входу.

## 23. Кешування

Завантаження однакових CSS, JavaScript і зображень для кожної сторінки витрачає трафік і час. Браузер може зберігати локальні копії.

Сервер повідомляє правила кешування через заголовки.

Поширені механізми:

- `Cache-Control`;
- `Expires`;
- `ETag`;
- `Last-Modified`;
- умовні запити;
- `304 Not Modified`.

Загальна схема перевірки:

```text
клієнт: у мене є версія з таким ETag
сервер: ресурс не змінився — 304
клієнт: використовує локальне тіло
```

У відповіді `304` повне тіло ресурсу не передається.

Кеш може існувати:

- у браузері;
- у reverse proxy;
- у CDN;
- у вебсервері;
- у серверному застосунку.

## 24. Стиснення

Текстові дані добре стискаються. Клієнт повідомляє підтримувані алгоритми:

```http
Accept-Encoding: gzip, br
```

Сервер може відповісти:

```http
Content-Encoding: gzip
```

Через мережу передається стиснене тіло, а клієнт розпаковує його.

Важливо розрізняти:

- `Content-Type` — що міститься в тілі;
- `Content-Encoding` — як тіло додатково перетворено для передавання.

Наприклад:

```http
Content-Type: text/html
Content-Encoding: gzip
```

означає: HTML-документ, стиснений gzip.

## 25. Повторне використання з’єднання

В HTTP/1.0 типовою була модель:

```text
TCP connect
→ request
→ response
→ close
```

Для кожного ресурсу доводилося створювати нове з’єднання.

В HTTP/1.1 з’єднання за замовчуванням може залишатися відкритим і використовуватися повторно:

```text
TCP connect
→ request/response
→ request/response
→ request/response
→ close
```

Це зменшує кількість службових обмінів і затримку.

HTTP/2 додав ефективніше паралельне передавання багатьох потоків у межах одного з’єднання.

HTTP/3 працює поверх QUIC, а не TCP, але зберігає семантику HTTP: методи, заголовки, URL і коди стану.

## 26. HTTP та HTTPS

Звичайний HTTP не шифрує дані. Учасник, який має можливість перехоплювати мережевий трафік, потенційно може прочитати або змінити повідомлення.

HTTPS — це HTTP, захищений TLS.

TLS забезпечує:

- конфіденційність;
- цілісність;
- автентифікацію сервера.

Перед обміном виконується рукостискання:

1. узгоджуються параметри шифрування;
2. сервер надає сертифікат;
3. клієнт перевіряє домен і ланцюжок довіри;
4. сторони формують ключі сеансу;
5. подальші дані шифруються.

Сертифікат допомагає переконатися, що клієнт встановив з’єднання із сервером потрібного домену, а не з підставним посередником.

HTTPS не гарантує, що сам сайт є чесним або не містить вразливостей. Він захищає канал і підтверджує володіння відповідним сертифікатом.

## 27. Проксі-сервери

Проксі є проміжною ланкою між клієнтом і сервером.

### 27.1. Forward proxy

Працює з боку клієнта:

```text
клієнт → forward proxy → зовнішній сервер
```

Може використовуватися для:

- контролю доступу;
- кешування;
- приховування безпосередньої адреси клієнта;
- корпоративної фільтрації.

### 27.2. Reverse proxy

Працює перед серверним застосунком:

```text
клієнт → reverse proxy → application server
```

Може виконувати:

- завершення TLS;
- балансування навантаження;
- кешування;
- стиснення;
- видачу статичних файлів;
- маршрутизацію;
- обмеження частоти запитів.

Приклад архітектури:

```text
браузер
  ↓
Nginx або Apache
  ├── CSS, JS, зображення
  └── запити бізнес-логіки
          ↓
      PHP / Python / Java / Node.js
          ↓
        база даних
```

Якщо reverse proxy не може отримати коректну відповідь від application server, клієнт може побачити `502 Bad Gateway`.

## 28. Балансування навантаження

Один публічний сервер може розподіляти запити між кількома внутрішніми екземплярами:

```text
               ┌→ application server 1
клієнт → proxy ├→ application server 2
               └→ application server 3
```

Послідовні запити одного користувача можуть обробляти різні сервери. Тому сесійний стан часто зберігають:

- у спільній базі даних;
- у Redis;
- у спеціальному сховищі сесій;
- у підписаному токені.

## 30. `curl`

`curl` — утиліта командного рядка та відома бібліотека для мережевих запитів.

Простий запит:

```bash
curl http://localhost:8000/
```

Заголовки відповіді:

```bash
curl -i http://localhost:8000/
```

Докладний обмін:

```bash
curl -v http://localhost:8000/
```

POST із JSON:

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -d '{"name":"Anna"}' \
  http://localhost:8000/api/users
```

Власний заголовок:

```bash
curl -H "Accept: application/json" \
  http://localhost:8000/api/users
```

`curl` особливо зручний для API, оскільки не намагається відображати відповідь як вебсторінку.

## 31. Найпростіший сервер на Python

```python
from http.server import BaseHTTPRequestHandler, HTTPServer


class Handler(BaseHTTPRequestHandler):
    def do_GET(self) -> None:
        if self.path == "/hello":
            body = "<h1>Hello!</h1>".encode("utf-8")

            self.send_response(200)
            self.send_header("Content-Type", "text/html; charset=UTF-8")
            self.send_header("Content-Length", str(len(body)))
            self.end_headers()
            self.wfile.write(body)
            return

        self.send_error(404, "Not Found")


server = HTTPServer(("127.0.0.1", 8000), Handler)
server.serve_forever()
```

Перевірка:

```text
http://localhost:8000/hello
```

або:

```bash
curl -v http://localhost:8000/hello
```

Сервер:

1. слухає порт `8000`;
2. отримує `GET`;
3. перевіряє шлях;
4. формує статус;
5. встановлює заголовки;
6. записує тіло.

Якщо змінити:

```python
self.send_header("Content-Type", "text/plain; charset=UTF-8")
```

браузер покаже `<h1>Hello!</h1>` як звичайний текст.

## 32. Статична й динамічна відповідь

Статичний вебсервер читає готовий файл:

```text
/index.html → файл index.html
```

Динамічний сервер формує тіло програмно:

```text
/profile → дані користувача → HTML або JSON
```

З точки зору клієнта обидва варіанти є HTTP-відповіддю. Браузеру не важливо, чи тіло було прочитане з диска, сформоване шаблонізатором або отримане з іншого сервісу.

## 33. Поширені помилки в розумінні HTTP

### 33.1. «HTTP і TCP — те саме»

TCP передає байти. HTTP визначає зміст і структуру прикладних повідомлень.

### 33.2. «HTTP використовується лише браузером»

Його використовують практично будь-які мережеві програми та API.

### 33.3. «POST автоматично безпечний»

Тіло POST не видно в адресному рядку, але без HTTPS воно не зашифроване.

### 33.4. «Base64 шифрує дані»

Base64 лише змінює представлення.

### 33.5. «401 означає заборонено»

`401` вимагає автентифікації, а `403` означає відмову в доступі.

### 33.6. «204 означає, що ресурс не змінився»

Для перевірки кешу використовується `304 Not Modified`.

### 33.7. «Cookie і сесія — те саме»

Cookie зберігається на клієнті. Серверна сесія є даними на сервері, а cookie часто містить лише її ідентифікатор.

### 33.8. «HTTPS захищає від усіх атак»

HTTPS захищає транспортний канал, але не виправляє помилки авторизації, SQL-ін’єкції, XSS або небезпечну бізнес-логіку.

## 34. Підсумки

HTTP є протоколом прикладного рівня, який структурує обмін даними між клієнтом і сервером.

Основна схема:

```text
TCP або QUIC
→ HTTP-запит
→ метод + URL + заголовки + тіло
→ серверна обробка
→ код стану + заголовки + тіло
→ клієнтська обробка
```

Ключові властивості:

- модель «запит — відповідь»;
- відсутність вбудованого стану між запитами;
- метаінформація в заголовках;
- коди стану;
- підтримка різних типів вмісту;
- кешування;
- стиснення;
- cookie та сесії;
- захищений транспорт через HTTPS;
- можливість використання проксі й розподіленої серверної архітектури.

Розуміння HTTP дозволяє не лише створювати вебсторінки, а й налагоджувати API, аналізувати помилки, пояснювати поведінку браузера та проєктувати взаємодію між компонентами програмної системи.
