# Лекція 6. Протокол HTTP

## Тема лекції

**Принципи роботи HTTP: структура запитів і відповідей, методи, заголовки, коди стану, сесії, кешування та захищене з’єднання.**

## Мета лекції

Сформувати розуміння того, як клієнт і сервер обмінюються даними через HTTP, навчити читати HTTP-повідомлення, пояснювати призначення основних методів, заголовків і кодів стану та аналізувати мережеву взаємодію за допомогою браузерних інструментів і `curl`.

## Очікувані результати навчання

Після лекції студент повинен уміти:

- пояснити відмінність між TCP і HTTP;
- описати роль сокета та мережевого з’єднання;
- пояснити модель «запит — відповідь»;
- прочитати структуру HTTP-запиту й відповіді;
- розрізняти стартовий рядок, заголовки та тіло;
- пояснити призначення основних HTTP-методів;
- інтерпретувати коди стану;
- пояснити роль заголовків `Host`, `Content-Type`, `Content-Length`, `Accept` і `Cookie`;
- пояснити відсутність стану в HTTP;
- описати механізм cookie й серверної сесії;
- пояснити принципи кешування та стиснення;
- розрізняти HTTP і HTTPS;
- пояснити призначення проксі-сервера;
- дослідити запит у вкладці Network;
- виконати запит через `curl`;
- створити простий локальний HTTP-сервер.

## План лекції

### 1. Мережеві протоколи та TCP

1. Протокол як набір узгоджених правил.
2. Передавання даних між двома програмами.
3. TCP як надійний двосторонній потік байтів.
4. Сокет як програмний інтерфейс мережевого з’єднання.
5. Аналогія між роботою з файлом і роботою із сокетом.
6. Межі відповідальності TCP.
7. Необхідність протоколу вищого рівня.

### 2. Призначення HTTP

1. HTTP як протокол прикладного рівня.
2. Початкове призначення — передавання гіпертекстових документів.
3. Передавання метаінформації разом із документом.
4. Тип, довжина й кодування вмісту.
5. Використання HTTP для API та взаємодії між програмами.
6. HTTP-клієнт не обов’язково є браузером.
7. Готові бібліотеки для роботи з HTTP.

### 3. Модель «запит — відповідь»

1. Клієнт ініціює запит.
2. Сервер формує відповідь.
3. Одна пара request/response як завершена взаємодія.
4. Одностороння ініціація в класичній моделі.
5. Незалежність окремих запитів.
6. Відсутність стану.
7. Необхідність додаткових механізмів для тривалої взаємодії.

### 4. HTTP поверх TCP

1. Встановлення TCP-з’єднання.
2. Надсилання текстового HTTP-повідомлення.
3. Отримання відповіді.
4. Закриття або повторне використання з’єднання.
5. Вартість встановлення нового з’єднання.
6. Постійні з’єднання.
7. `Connection: keep-alive`.
8. Відмінності HTTP/1.0 і HTTP/1.1.
9. Загальне уявлення про HTTP/2 та HTTP/3.

### 5. Структура HTTP-запиту

1. Стартовий рядок.
2. Метод.
3. Ціль запиту.
4. Версія протоколу.
5. Заголовки.
6. Порожній рядок.
7. Тіло запиту.
8. Приклад сирого запиту.

```http

```

1. Рядок статусу.

## Рекомендовані демонстрації

Покроковий сценарій із командами запуску: [`6.demo_guide.md`](6.demo_guide.md). Код: [`src/`](src/).

1. Відкрити вкладку Network і перезавантажити сторінку.
2. Показати окремі запити до HTML, CSS, JavaScript і зображень.
3. Розібрати стартовий рядок і заголовки запиту.
4. Розібрати код стану й заголовки відповіді.
5. Показати `Content-Type`.
6. Повернути той самий вміст як `text/plain` і `text/html` (`src/1.simple_http_server`).
7. Показати cookie у DevTools.
8. Видалити сесійне cookie та продемонструвати втрату авторизації.
9. Знайти відповідь `304 Not Modified`.
10. Виконати `curl -v` проти локального сервера.
11. Надіслати власний заголовок через `curl` (`src/2.flask_routes`, `X-token`).
12. Запустити простий Python HTTP-сервер (`src/1.simple_http_server`).
13. Програмно сформувати відповідь `200 OK`.
14. Продемонструвати `404 Not Found`.
15. Показати програмний клієнт без браузера (`src/4.http_client_python`).
16. (Опційно) JSON API (`src/3.rest_api`) та зв’язок з AJAX у L4.
17. Показати схему reverse proxy → application server.
