# Лекція 2. HTML та розгортання вебсайту в локальному середовищі

## 1. HTML у системі вебтехнологій

Клієнтська частина сучасного вебзастосунку зазвичай базується на трьох основних технологіях:

- **HTML** описує структуру та зміст сторінки;
- **CSS** визначає її зовнішнє оформлення;
- **JavaScript** додає поведінку та інтерактивність.

HTML розшифровується як **HyperText Markup Language** — мова розмітки гіпертексту. HTML не є мовою програмування: він не призначений для опису алгоритмів, обчислень, умов і циклів. Його завдання — позначити логічні частини документа та зв’язки між ними.

За допомогою HTML можна вказати, що певний фрагмент є заголовком, абзацом, списком, посиланням, зображенням, таблицею або формою. Браузер читає цей опис, будує внутрішнє представлення документа й відображає сторінку.

HTML виник разом із раннім Всесвітнім павутинням. У різні періоди браузери додавали власні теги й атрибути, через що сторінки могли по-різному працювати в різних програмах. Поступова стандартизація привела до HTML5. Сьогодні HTML розвивається як живий стандарт без регулярного випуску нових великих номерів версій.

## 2. HTML-документ як текстовий файл

HTML-документ є звичайним текстовим файлом із розширенням `.html` або `.htm`. Його можна відкрити у браузері та в текстовому редакторі.

```html
<h1>Основи вебтехнологій</h1>
<p>Це приклад HTML-документа.</p>
```

У редакторі видно вихідну розмітку, а браузер показує її як заголовок і абзац. Для роботи з HTML зручно використовувати редактори з підсвічуванням синтаксису, наприклад Visual Studio Code, Notepad++ або WebStorm.

Браузери історично намагаються відобразити сторінку навіть за наявності помилок: пропущених закривальних тегів, неправильної вкладеності або неповної структури. Це потрібно для сумісності зі старими сайтами. Проте покладатися на автоматичне виправлення не слід: браузер може побудувати DOM не так, як очікував автор.

## 3. Мінімальна структура HTML-документа

```html
<!DOCTYPE html>
<html lang="uk">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Моя перша сторінка</title>
</head>
<body>
    <h1>Вітаємо!</h1>
    <p>Це локальна HTML-сторінка.</p>
</body>
</html>
```

### 3.1. Декларація `DOCTYPE`

```html
<!DOCTYPE html>
```

Декларація повідомляє браузеру, що документ потрібно обробляти відповідно до сучасного стандарту HTML. Вона не створює видимого елемента. Без коректного `DOCTYPE` браузер може перейти в режим сумісності зі старими сторінками.

### 3.2. Кореневий елемент `<html>`

Елемент `<html>` містить увесь документ. Атрибут `lang="uk"` указує основну мову сторінки. Це корисно для пошукових систем, перекладачів, перевірки правопису та програм екранного читання.

### 3.3. Секція `<head>`

У `<head>` розміщують службову інформацію:

- кодування;
- заголовок вкладки;
- метадані;
- посилання на таблиці стилів;
- сценарії;
- параметри для мобільних пристроїв;
- інформацію для пошукових систем.

Вміст `<head>` зазвичай не відображається як частина сторінки.

### 3.4. Секція `<body>`

У `<body>` розміщують основний видимий вміст: текст, посилання, зображення, списки, таблиці, форми та інші елементи інтерфейсу.

### 3.5. Кодування

```html
<meta charset="UTF-8">
```

UTF-8 підтримує українські символи та більшість писемностей світу. Без явного зазначення кодування браузер може неправильно інтерпретувати текст.

## 4. Теги, елементи та атрибути

HTML-елемент у найпростішому випадку складається з відкривального тегу, вмісту та закривального тегу:

```html
<p>Текст абзацу</p>
```

- `<p>` — відкривальний тег;
- `Текст абзацу` — вміст;
- `</p>` — закривальний тег;
- уся конструкція — HTML-елемент.

Більшість елементів є парними. Деякі елементи не містять вкладеного вмісту й не мають окремого закривального тегу:

```html
<img src="photo.jpg" alt="Опис фотографії">
<br>
<meta charset="UTF-8">
```

Атрибути задають додаткові властивості:

```html
<a href="https://example.com" target="_blank">Відкрити сайт</a>
```

Тут `href` визначає адресу, а `target="_blank"` — відкриття в новій вкладці.

Поширені глобальні атрибути:

- `id` — унікальний ідентифікатор;
- `class` — належність до класу або кількох класів;
- `title` — додаткова підказка;
- `hidden` — приховування елемента;
- `data-*` — користувацькі дані для сценаріїв.

Елементи можуть вкладатися один в один:

```html
<section>
    <h2>Новини</h2>
    <p>Текст новини.</p>
</section>
```

Закривати вкладені елементи слід у зворотному порядку:

```html
<p><strong>Правильна вкладеність</strong></p>
```

## 5. Спеціальні символи та HTML-сутності

Символи `<`, `>` і `&` мають службове значення. Для їх відображення як тексту використовують HTML-сутності:

```html
<p>Тег абзацу записується як &lt;p&gt;.</p>
<p>Символ амперсанда: &amp;</p>
```

| Результат | Запис |
|---|---|
| `<` | `&lt;` |
| `>` | `&gt;` |
| `&` | `&amp;` |
| нерозривний пробіл | `&nbsp;` |

`&nbsp;` не дозволяє браузеру переносити рядок у цьому місці. Для створення відступів потрібно використовувати CSS, а не набір нерозривних пробілів.

## 6. Блокові та рядкові елементи

Блоковий елемент зазвичай починається з нового рядка й займає доступну ширину контейнера. Приклади:

- `<div>`;
- `<p>`;
- `<h1>`–`<h6>`;
- `<section>`;
- `<article>`;
- `<header>`;
- `<footer>`;
- `<ul>` і `<ol>`.

Рядкові елементи розміщуються всередині текстового рядка й займають лише потрібну ширину:

- `<span>`;
- `<a>`;
- `<strong>`;
- `<em>`;
- `<code>`.

Зображення `<img>` має власні ширину й висоту, але за замовчуванням вбудовується в текстовий рядок.

`<div>` є універсальним блоковим контейнером, а `<span>` — універсальним рядковим контейнером. Самі по собі вони не надають вмісту спеціального змісту, але зручні для групування, стилізації та керування через JavaScript.

## 7. Семантика HTML

Семантика означає зміст і призначення елемента. Наприклад:

- `<nav>` позначає навігацію;
- `<main>` — головний вміст;
- `<article>` — самостійний матеріал;
- `<section>` — тематичний розділ;
- `<header>` — вступну або заголовкову частину;
- `<footer>` — завершальну частину;
- `<button>` — кнопку.

Семантичні теги можуть виглядати так само, як звичайні контейнери, але роблять структуру зрозумілішою для розробників, пошукових систем, програм екранного читання та режимів читання. Тому не варто будувати всю сторінку лише з `<div>`, якщо існують точніші елементи.

## 8. Основні елементи сторінки

### 8.1. Заголовки та абзаци

```html
<h1>Назва сторінки</h1>
<h2>Назва розділу</h2>
<p>Текст абзацу.</p>
```

Підрівні заголовків слід використовувати відповідно до структури документа, а не лише через бажаний розмір шрифту.

### 8.2. Списки

```html
<ul>
    <li>HTML</li>
    <li>CSS</li>
    <li>JavaScript</li>
</ul>
```

```html
<ol>
    <li>Створити файл.</li>
    <li>Додати розмітку.</li>
    <li>Відкрити сторінку.</li>
</ol>
```

### 8.3. Посилання та якорі

```html
<a href="about.html">Про сайт</a>
```

Вмістом посилання може бути не лише текст, а й зображення або група елементів.

Для переходу до частини тієї самої сторінки використовують якір:

```html
<a href="#contacts">Перейти до контактів</a>

<section id="contacts">
    <h2>Контакти</h2>
</section>
```

### 8.4. Зображення

```html
<img src="images/logo.png" alt="Логотип кафедри">
```

Атрибут `alt` містить текстовий опис. Він потрібний, якщо зображення не завантажилося, а також для доступності.

### 8.5. Таблиці

Таблиці призначені для табличних даних, а не для побудови макета сторінки:

```html
<table>
    <thead>
        <tr>
            <th>Технологія</th>
            <th>Призначення</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <td>HTML</td>
            <td>Структура</td>
        </tr>
        <tr>
            <td>CSS</td>
            <td>Оформлення</td>
        </tr>
    </tbody>
</table>
```

Історично таблиці застосовували для розміщення елементів сторінки, але сучасні макети слід створювати засобами CSS, зокрема Flexbox і Grid.

## 9. DOM — об’єктна модель документа

Після завантаження HTML браузер аналізує текст і будує **DOM-дерево** — Document Object Model.

Для документа:

```html
<body>
    <h1>Заголовок</h1>
    <p>Абзац</p>
</body>
```

елемент `<body>` є батьківським вузлом для `<h1>` і `<p>`. У складнішій сторінці утворюється дерево з великою кількістю вкладених вузлів.

DOM важливий тому, що браузер використовує його для побудови сторінки, а JavaScript може знаходити, створювати, змінювати й видаляти елементи. Саме можливість змінювати DOM перетворила статичні сторінки на інтерактивні інтерфейси.

## 10. Підключення CSS і JavaScript

Таблицю стилів зазвичай підключають у `<head>`:

```html
<link rel="stylesheet" href="css/style.css">
```

Сценарій можна підключити так:

```html
<script src="js/main.js" defer></script>
```

Класичний сценарій без додаткових атрибутів може призупиняти обробку HTML, доки файл не буде завантажено та виконано. Через це великий сценарій на початку документа може затримати відображення сторінки.

`defer` дозволяє завантажувати сценарій паралельно з аналізом HTML і виконати його після побудови документа. `async` також завантажує файл паралельно, але виконує його відразу після готовності, тому порядок виконання кількох сценаріїв не гарантується.

## 11. HTML-форми

Форми дозволяють користувачу ввести дані й відправити їх на сервер:

```html
<form action="/feedback" method="post">
    <label for="name">Ім’я:</label>
    <input id="name" name="name" type="text" required>

    <label for="message">Повідомлення:</label>
    <textarea id="message" name="message"></textarea>

    <button type="submit">Надіслати</button>
</form>
```

Важливі атрибути:

- `action` — адреса обробника;
- `method` — HTTP-метод;
- `name` — ім’я параметра;
- `value` — значення;
- `type` — тип поля;
- `required` — вимога заповнити поле.

Поле без `name` зазвичай не включається до даних форми.

### 11.1. Метод `GET`

Під час використання `GET` дані додаються до URL:

```text
/search?query=html&page=1
```

Це зручно для пошуку, фільтрації та сторінок, адресу яких потрібно зберегти або передати. Дані видно в адресному рядку, тому `GET` не слід використовувати для паролів та іншої чутливої інформації.

### 11.2. Метод `POST`

Під час використання `POST` дані передаються в тілі HTTP-запиту. Метод зручний для створення або зміни ресурсів, великих форм і завантаження файлів.

Наявність даних у тілі запиту не робить їх автоматично зашифрованими. Конфіденційність забезпечує HTTPS.

### 11.3. URL-кодування

Не всі символи можна безпосередньо записати в URL. Пробіли, національні символи та службові знаки кодуються у вигляді послідовностей `%...`. Браузер може показувати адресу у зручному вигляді, але фактичний запит містить закодовані байти.

### 11.4. Приховані поля та стан

```html
<input type="hidden" name="order_id" value="125">
```

Приховане поле не відображається користувачу, але надсилається разом із формою. Воно може передавати ідентифікатор об’єкта або інший контекст.

HTTP не зберігає стан автоматично: кожен запит є окремою взаємодією. Контекст передається через URL, поля форми, cookie, заголовки, токени або серверні сесії.

Приховані поля не можна вважати захищеними. Користувач може змінити їх через DevTools, тому сервер зобов’язаний перевіряти всі отримані дані.

## 12. Інструменти розробника браузера

DevTools відкриваються клавішею `F12`, комбінацією `Ctrl+Shift+I` або через команду «Дослідити елемент».

### 12.1. Elements або Inspector

Вкладка показує DOM-дерево. Під час наведення браузер підсвічує відповідну область сторінки. Тут можна:

- перевірити вкладеність;
- знайти зайвий контейнер;
- побачити класи й атрибути;
- тимчасово змінити текст;
- додати або видалити елемент;
- переглянути застосовані CSS-правила.

Зміни в DevTools не редагують вихідний файл і зникнуть після перезавантаження.

### 12.2. Box Model

Для вибраного елемента показуються:

- content;
- padding;
- border;
- margin.

Це допомагає зрозуміти, чому елемент має певний розмір або відступ.

### 12.3. Console

У консолі відображаються повідомлення, попередження, помилки JavaScript і результати виконання команд. Якщо сценарій перестав працювати, Console є одним із перших місць для перевірки.

### 12.4. Network

Вкладка Network показує:

- URL;
- HTTP-метод;
- код стану;
- час виконання;
- розмір відповіді;
- заголовки запиту й відповіді;
- параметри форми;
- отримані HTML, CSS, JavaScript, зображення та інші ресурси.

Вона дозволяє побачити, чи справді браузер звернувся до сервера, який код стану отримав і який MIME-тип повернув сервер.

## 13. Відкриття HTML-файлу через `file://`

Найпростіший спосіб переглянути сторінку — відкрити `index.html` у браузері. Адреса матиме приблизно такий вигляд:

```text
file:///C:/projects/my-site/index.html
```

Для простої статичної сторінки цього може бути достатньо. Проте сторінка, відкрита через `file://`, не є повним аналогом сайта, отриманого від вебсервера.

Можливі обмеження:

- немає HTTP-заголовків;
- відсутні справжні HTTP-запити до поточного сайта;
- інакше визначається походження ресурсу;
- можуть блокуватися `fetch`, модулі JavaScript і доступ до локальних файлів;
- не працює серверна логіка;
- не можна коректно перевірити маршрутизацію;
- відсутні серверні коди стану та налаштування MIME-типів.

Причиною є політика безпеки браузера. Локальні файли не повинні отримувати довільний доступ до інших файлів комп’ютера.

Тому `file://` підходить для першого знайомства з HTML, але реальну розробку краще виконувати через локальний HTTP-сервер.

## 14. Локальний вебсервер

Вебсервер — програма, яка:

1. слухає мережевий порт;
2. приймає HTTP-запити;
3. знаходить або генерує ресурс;
4. повертає HTTP-відповідь.

Для локальної розробки сервер працює на тому самому комп’ютері, що й браузер.

### 14.1. `localhost` і `127.0.0.1`

Ім’я `localhost` зазвичай указує на поточний комп’ютер. Для IPv4 йому відповідає адреса `127.0.0.1`.

```text
http://localhost:8000/
```

Ця адреса означає: використати HTTP, звернутися до поточного комп’ютера, підключитися до порту `8000` і запросити кореневий ресурс `/`.

### 14.2. Порти

На одному комп’ютері може працювати багато мережевих програм. Порт визначає, якій програмі призначене з’єднання.

Типові порти:

- `80` — HTTP;
- `443` — HTTPS;
- `8000`, `8080`, `3000`, `5173` — поширені порти локальної розробки.

Якщо використовується стандартний порт `80`, його можна не вказувати:

```text
http://localhost/
```

## 15. Найпростіші способи запуску сайта

### 15.1. Вбудований сервер Python

У каталозі сайта можна виконати:

```bash
python -m http.server 8000
```

Після цього сторінка буде доступна за адресою:

```text
http://localhost:8000/
```

Цей сервер зручний для статичних файлів, але не є виробничим сервером і не виконує PHP-код.

### 15.2. Live Server

Розширення Live Server для Visual Studio Code запускає простий локальний сервер і зазвичай автоматично оновлює сторінку після збереження файлів. Це зручно для навчальних статичних проєктів, але важливо розуміти, що розширення лише автоматизує запуск HTTP-сервера.

### 15.3. Сервери фреймворків

Сучасні інструменти часто мають власний сервер розробки. Він може не лише роздавати файли, а й виконувати складання модулів, автоматичне оновлення та перетворення ресурсів.

## 16. Apache у локальному середовищі

Apache HTTP Server — один із найвідоміших вебсерверів. Його можна встановити у Windows, Linux або в Linux-середовищі WSL.

Загальна послідовність:

1. встановити Apache;
2. визначити кореневий каталог сайта;
3. розмістити в ньому `index.html`;
4. запустити службу;
5. відкрити `http://localhost`.

У Linux типовим кореневим каталогом може бути:

```text
/var/www/html
```

Конкретний шлях залежить від конфігурації.

`index.html` часто є файлом за замовчуванням. Коли браузер запитує каталог `/`, сервер шукає індексний документ і повертає його без явного зазначення імені файла.

Apache може також працювати з PHP, проксіювати запити до іншого застосунку, обслуговувати віртуальні хости, HTTPS і правила маршрутизації.

## 17. WSL як середовище веброзробки

WSL дозволяє запускати Linux-середовище всередині Windows. Це зручно, оскільки значна частина вебсерверів у реальних системах працює під Linux.

У WSL можна:

- встановити Apache або Nginx;
- працювати з Linux-командами;
- запускати PHP, Python і бази даних;
- наблизити локальне середовище до серверного.

Windows і WSL є різними мережевими середовищами, хоча сучасна інтеграція часто дозволяє звертатися до сервісу WSL через `localhost`.

У разі проблем потрібно перевірити:

- чи запущений сервер;
- на якій адресі він слухає;
- який порт використовується;
- чи не блокує підключення брандмауер;
- яку IP-адресу має WSL;
- чи працює перенаправлення портів.

Усередині кожної віртуальної машини або контейнера `localhost` указує саме на це середовище, а не автоматично на хост-комп’ютер.

## 18. Docker для локального розгортання

Docker запускає застосунки в контейнерах. Контейнер нагадує легке ізольоване середовище, яке містить лише необхідні компоненти.

Переваги Docker:

- однакове середовище на різних комп’ютерах;
- автоматизоване встановлення залежностей;
- ізоляція версій;
- швидкий запуск і видалення;
- зручне перенесення конфігурації;
- запуск кількох пов’язаних сервісів.

### 18.1. Статичний сайт у контейнері Nginx

```bash
docker run --rm \
    -p 8080:80 \
    -v "$PWD:/usr/share/nginx/html:ro" \
    nginx:alpine
```

Після запуску сайт доступний за адресою:

```text
http://localhost:8080/
```

`-p 8080:80` означає: спрямувати порт `8080` хост-комп’ютера на порт `80` усередині контейнера.

`-v` монтує локальний каталог у каталог, з якого Nginx роздає файли. У Windows синтаксис шляху може відрізнятися.

### 18.2. Власний Docker-образ

```dockerfile
FROM nginx:alpine
COPY . /usr/share/nginx/html
```

Побудова образу:

```bash
docker build -t lecture-site .
```

Запуск:

```bash
docker run --rm -p 8080:80 lecture-site
```

Тепер образ містить і вебсервер, і файли сайта.

### 18.3. Docker Compose

Реальний вебзастосунок часто складається з кількох компонентів:

- вебсервер;
- PHP або Python-застосунок;
- база даних;
- кеш;
- фоновий обробник;
- додаткова служба.

Docker Compose дозволяє описати їх в одному файлі та запускати як систему:

```yaml
services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"
    volumes:
      - ./site:/usr/share/nginx/html:ro
```

Запуск:

```bash
docker compose up
```

Зупинка:

```bash
docker compose down
```

Головна перевага полягає не лише в запуску сайта. У конфігурації зберігається опис середовища, який можна відтворити на іншому комп’ютері.

## 19. Від локальної сторінки до локального сайта

Рекомендована послідовність роботи:

1. Створити каталог проєкту.
2. Додати `index.html`.
3. Побудувати коректну структуру документа.
4. Додати текст, посилання, зображення та форму.
5. Відкрити файл через `file://`.
6. Перевірити DOM у DevTools.
7. Запустити локальний HTTP-сервер.
8. Відкрити сайт через `http://localhost`.
9. Перевірити запити у вкладці Network.
10. За потреби перенести сайт у WSL або Docker.

Приклад структури:

```text
my-site/
├── index.html
├── about.html
├── css/
│   └── style.css
├── js/
│   └── main.js
└── images/
    └── logo.png
```

Для ресурсів варто використовувати відносні шляхи:

```html
<link rel="stylesheet" href="css/style.css">
<script src="js/main.js" defer></script>
<img src="images/logo.png" alt="Логотип">
<a href="about.html">Про сайт</a>
```

Так проєкт легше переносити між каталогами, серверами й контейнерами.

## 20. Поширені помилки

### 20.1. HTML називають мовою програмування

HTML описує структуру документа. Алгоритмічна поведінка реалізується JavaScript або серверною мовою.

### 20.2. Використовують HTML для оформлення

Колір, відступи, шрифти й макет потрібно переважно задавати в CSS. HTML має зберігати структуру та семантику.

### 20.3. Не зазначають кодування

Без UTF-8 український текст може відображатися некоректно.

### 20.4. Ігнорують `alt`

Без альтернативного тексту погіршується доступність, а користувач не отримує пояснення, якщо зображення не завантажилося.

### 20.5. Перевіряють складний сайт лише через `file://`

Модулі, HTTP-запити, маршрутизація та серверні функції можуть не працювати або поводитися інакше.

### 20.6. Плутають порти хоста й контейнера

У записі `8080:80` браузер звертається до порту `8080` хоста, а Docker передає трафік на порт `80` контейнера.

### 20.7. Вважають дані форми надійними

Будь-які значення з браузера можуть бути змінені користувачем. Перевірка на клієнті покращує інтерфейс, але сервер повинен повторно перевіряти дані.

## 21. Підсумки

HTML є основою структури вебсторінки. Він описує логічні елементи документа, їх вкладеність, атрибути та семантику. Після отримання HTML браузер будує DOM-дерево, застосовує стилі, завантажує ресурси та виконує сценарії.

Для дослідження сторінки використовуються DevTools. Вкладка Elements показує DOM і стилі, Console — помилки й повідомлення JavaScript, Network — HTTP-запити та відповіді.

Просту сторінку можна відкрити безпосередньо з диска, але схема `file://` має обмеження й не відтворює справжню роботу сайта. Для повноцінної локальної розробки потрібно запускати HTTP-сервер.

Найпростіші варіанти — Live Server або вбудований сервер Python. Apache та WSL дозволяють працювати в середовищі, близькому до традиційного Linux-сервера. Docker забезпечує ізоляцію, автоматизацію й відтворюваність, а Docker Compose дає змогу запускати багатокомпонентні вебзастосунки.

Загальна послідовність практичної роботи:

```text
HTML-файл → браузер → DevTools → локальний HTTP-сервер
→ відтворюване середовище WSL або Docker
```
