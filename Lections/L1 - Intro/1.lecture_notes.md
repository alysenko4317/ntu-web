# Конспект лекції 1  
## Вступ до вебтехнологій і принципів роботи вебзастосунків

## 1. Вступ

Вебтехнології сьогодні використовуються значно ширше, ніж лише для створення традиційних сайтів. На їхній основі працюють інтернет-магазини, системи електронного навчання, банківські сервіси, соціальні мережі, поштові системи, панелі керування мережевим обладнанням, хмарні застосунки та велика кількість внутрішніх корпоративних систем.

Мета курсу полягає не лише у вивченні окремих мов або бібліотек. Важливо зрозуміти загальні принципи:

- як побудований вебзастосунок;
- як клієнт взаємодіє із сервером;
- як передаються дані мережею;
- як сервер обробляє запит;
- як браузер визначає, що саме він отримав;
- як пов’язані HTML, CSS, JavaScript, серверне програмування та бази даних.

Курс передбачає наявність базових навичок програмування. Для серверних прикладів може використовуватися Python, оскільки він має відносно простий синтаксис і зручні вебфреймворки. Водночас принципи серверної розробки не залежать від конкретної мови: аналогічні завдання можна розв’язувати за допомогою PHP, Java, C#, JavaScript/Node.js та інших технологій.

Вебдизайн є окремою професійною сферою. Він охоплює композицію, кольори, типографіку, зручність інтерфейсу та спонукання користувача до певних дій. У цьому курсі основна увага приділяється технічній побудові вебсистем.

Сучасні системи штучного інтелекту можуть допомагати створювати розмітку, стилі, програмний код і навіть початкові варіанти дизайну. Проте для перевірки та коректного використання згенерованого результату все одно потрібно розуміти базові принципи роботи вебу.

## 2. Що таке вебзастосунок

У найпростішому випадку вебзастосунок складається з двох основних частин:

1. **Клієнт** — програма, яка надсилає запит і обробляє отриману відповідь.
2. **Сервер** — програма, яка приймає запит, виконує потрібну логіку та повертає відповідь.

Типовим вебклієнтом є браузер. Проте клієнтом може бути також:

- мобільний застосунок;
- настільна програма;
- Telegram-бот;
- консольна утиліта;
- програмна бібліотека;
- інший сервер.

Отже, вебтехнології не слід ототожнювати лише зі сторінкою, яку людина бачить у браузері. У технічному сенсі головною є мережева взаємодія за певними правилами.

### 2.1. Frontend

**Frontend** — частина системи, з якою безпосередньо взаємодіє користувач. У веббраузері вона зазвичай будується за допомогою трьох базових технологій:

- **HTML** задає структуру документа;
- **CSS** визначає зовнішній вигляд;
- **JavaScript** реалізує програмну поведінку.

HTML є мовою розмітки, а не повноцінною мовою програмування. Він описує логічну структуру сторінки: заголовки, абзаци, списки, таблиці, форми, зображення, посилання тощо.

CSS визначає, як ці елементи повинні виглядати: колір, шрифт, розмір, відступи, розташування, адаптацію до різних екранів.

JavaScript дає змогу реагувати на дії користувача, змінювати документ після завантаження, перевіряти дані форм, надсилати додаткові запити до сервера та оновлювати частини інтерфейсу.

### 2.2. Backend

**Backend** — серверна частина системи. Вона може:

- приймати й перевіряти запити;
- виконувати бізнес-логіку;
- працювати з базою даних;
- перевіряти права доступу;
- взаємодіяти з іншими сервісами;
- формувати HTML-документи;
- повертати структуровані дані, наприклад JSON;
- зберігати завантажені файли;
- вести журнал подій.

Backend може бути реалізований практично будь-якою мовою програмування. Вибір залежить від вимог проєкту, команди, продуктивності, наявних бібліотек та інфраструктури.

### 2.3. Full-stack

**Full-stack розробник** працює і з клієнтською, і з серверною частиною. Іноді обидві частини реалізують за допомогою JavaScript: у браузері та на сервері через Node.js. Однак full-stack не означає обов’язкове використання однієї мови.

### 2.4. Інші значення frontend і backend

Ці терміни можуть використовуватися ширше. Frontend іноді означає зовнішній програмний інтерфейс, а backend — внутрішній компонент, який виконує основну роботу. Таке вживання можливе не лише у веброзробці.

## 3. Вебінтерфейс поза браузером

Вебтехнології часто застосовують для побудови інтерфейсів інших програм. Наприклад, сторінка налаштування маршрутизатора є невеликим вебзастосунком, який працює на самому пристрої.

У настільну або мобільну програму можна вбудувати компонент веббраузера та відображати в ньому інтерфейс, створений за допомогою HTML, CSS і JavaScript. Це дає змогу повторно використовувати вебтехнології у звичайних програмах.

Існують також інші мови розмітки інтерфейсів. Наприклад, у платформі .NET використовується XAML. Він не є HTML, але ґрунтується на подібній ідеї: структура інтерфейсу описується декларативно за допомогою елементів і властивостей.

## 4. Інтернет і World Wide Web

**Інтернет** — глобальна система взаємопов’язаних комп’ютерних мереж.

Він виник унаслідок поступового об’єднання окремих локальних мереж університетів, дослідницьких установ, підприємств та інших організацій. Інтернет є транспортною інфраструктурою, через яку можуть працювати різні сервіси.

**World Wide Web**, або WWW, — лише один із сервісів Інтернету. Він є системою взаємопов’язаних документів і ресурсів, доступ до яких зазвичай здійснюється за допомогою браузера та протоколу HTTP або HTTPS.

Крім WWW, в Інтернеті існують:

- електронна пошта;
- передавання файлів;
- віддалений доступ;
- служби імен;
- обмін повідомленнями;
- потокове передавання мультимедіа;
- численні спеціалізовані протоколи.

Важлива перевага вебу полягає у централізованому розміщенні інформації. Документ зберігається на сервері. Коли автор оновлює його, усі користувачі під час наступного звернення отримують актуальну версію. Не потрібно надсилати нову копію кожному користувачу окремо.

---

## 5. Вебсторінка, вебсайт і вебзастосунок

**Вебсторінка** — окремий документ або представлення ресурсу, яке може бути відкрито за певною адресою.

**Вебсайт** історично означав сукупність пов’язаних вебсторінок і ресурсів.

**Вебзастосунок** — система, у якій сторінки є лише частиною програмної взаємодії. Така система має логіку, обробляє дані, зберігає стан, взаємодіє з користувачем та іншими сервісами.

Чітка межа між сайтом і вебзастосунком сьогодні майже зникла. Інтернет-магазин, пошукова система або соціальна мережа формально можуть називатися сайтами, але фактично є складними програмними системами.

## 6. Еволюція вебтехнологій

## 6.1. Статичні сторінки

Перші вебресурси складалися переважно зі статичних HTML-файлів. Схема роботи була простою:

1. Користувач вводив адресу.
2. Браузер надсилав запит.
3. Вебсервер знаходив файл на диску.
4. Сервер передавав файл браузеру.
5. Браузер відображав документ.

Статичний документ однаковий для всіх користувачів. Щоб змінити його, потрібно відредагувати файл на сервері.

Такий підхід добре підходить для незмінної інформації, але має обмеження. Наприклад, статичний файл не може самостійно:

- виконати пошук у каталозі;
- показати персональні дані користувача;
- отримати дані з бази;
- обробити замовлення;
- змінити вміст залежно від параметрів запиту.

## 6.2. Серверні сценарії та CGI

Наступним кроком стала генерація документів програмою.

Замість того щоб завжди читати готовий файл із диска, вебсервер міг запускати зовнішню програму. Програма виконувала певні дії, наприклад зверталася до бази даних, а результат її роботи передавався клієнту.

Одним із ранніх механізмів був **CGI — Common Gateway Interface**. Сервер запускав сценарій, передавав йому параметри запиту й отримував сформовану відповідь.

Для таких сценаріїв використовували Perl та інші мови. Пізніше з’явилися PHP, Java Servlets, ASP, ASP.NET, Python-фреймворки та багато інших платформ.

Основна ідея залишається актуальною: документ або дані не обов’язково існують на диску у готовому вигляді. Сервер може сформувати їх під час обробки запиту.

## 6.3. Серверні шаблони

Поширеним підходом стала генерація HTML за допомогою шаблонів:

1. На сервері зберігається шаблон сторінки.
2. Серверна програма отримує дані з бази.
3. Дані підставляються у шаблон.
4. Готовий HTML надсилається браузеру.

Наприклад, один шаблон сторінки товару може використовуватися для тисяч товарів. Конкретна назва, ціна, опис і зображення підставляються динамічно.

## 6.4. Java у браузері та JavaScript

На ранніх етапах розвитку вебу робилися спроби запускати у браузері програми у вигляді Java-аплетів. Вони давали змогу створювати складнішу поведінку, але мали проблеми із сумісністю, швидкодією та безпекою. Згодом браузери припинили їх підтримувати.

JavaScript виявився зручнішим засобом програмування безпосередньо всередині вебсторінки. Він став стандартною мовою клієнтської частини.

Не слід плутати Java і JavaScript. Це різні мови з різною історією та сферами використання.

## 6.5. AJAX

Спочатку для отримання нових даних браузеру часто доводилося повністю перезавантажувати сторінку.

Наприклад, користувач обирає місто, після чого потрібно завантажити перелік відділень служби доставки. Без асинхронних запитів браузер мав би відправити форму, отримати новий HTML і повністю перемалювати сторінку.

**AJAX** дав змогу JavaScript-коду надсилати додатковий запит у фоновому режимі:

1. Користувач виконує дію.
2. JavaScript надсилає запит серверу.
3. Сервер повертає лише потрібні дані.
4. JavaScript оновлює окрему частину сторінки.

Повне перезавантаження не потрібне. Це значно покращило зручність вебінтерфейсів.

Назва AJAX історично пов’язана з XML, але сучасні застосунки частіше використовують JSON.

## 6.6. jQuery та сучасні фреймворки

У певний період дуже популярною стала бібліотека **jQuery**. Вона спрощувала:

- роботу з HTML-документом;
- обробку подій;
- анімацію;
- AJAX-запити;
- сумісність між браузерами.

У нових великих проєктах її значною мірою замінили сучасні бібліотеки та фреймворки, зокрема React і Angular. Однак jQuery досі можна зустріти у старих системах і невеликих проєктах.

## 6.7. Від готового HTML до API

У традиційній моделі сервер формує готовий HTML. У сучасних системах сервер часто повертає лише дані, наприклад у форматі JSON.

Клієнтська програма отримує ці дані та сама будує інтерфейс.

Переваги такого підходу:

- один сервер може обслуговувати різні типи клієнтів;
- браузер, мобільний застосунок і настільна програма можуть використовувати однаковий API;
- сервер не зобов’язаний знати, як саме дані будуть відображені;
- клієнтський інтерфейс може оновлюватися без повного перезавантаження сторінки.

**API** — програмний інтерфейс, за допомогою якого один компонент системи звертається до іншого.

## 6.8. Односторінкові застосунки

У SPA, або односторінковому застосунку, браузер спочатку отримує базовий HTML і JavaScript-код. Далі JavaScript:

- звертається до API;
- отримує дані;
- будує інтерфейс;
- перемикає представлення;
- оновлює лише потрібні частини сторінки.

З погляду користувача така система може поводитися подібно до настільної програми.

## 7. Клієнт-серверна модель

Клієнт і сервер повинні однаково розуміти структуру повідомлень. Для цього використовуються **протоколи**.

Протокол — це формалізований набір правил, який визначає:

- як встановлюється взаємодія;
- як виглядає запит;
- як виглядає відповідь;
- які службові дані передаються;
- як повідомляється про помилки;
- як інтерпретується вміст.

Для вебу основним протоколом прикладного рівня є HTTP.

У спрощеному вигляді взаємодія відбувається так:

```text
Клієнт → HTTP-запит → Сервер
Клієнт ← HTTP-відповідь ← Сервер
```

Сервер може повернути:

- HTML;
- CSS;
- JavaScript;
- зображення;
- аудіо або відео;
- PDF;
- JSON;
- звичайний текст;
- архів;
- повідомлення про помилку.

HTTP є протоколом прикладного рівня. Під ним працюють інші мережеві механізми, які забезпечують адресацію, доставку пакетів і передавання даних фізичною мережею. Детальніше це вивчається у курсах комп’ютерних мереж, але веброзробник повинен розуміти принаймні основні поняття: IP-адресу, DNS, порт і протокол.

## 8. HTTP-запит і відповідь

HTTP-запит містить:

- метод;
- адресу ресурсу;
- заголовки;
- іноді тіло запиту.

Наприклад, метод `GET` зазвичай використовується для отримання ресурсу.

HTTP-відповідь містить:

- статус;
- заголовки;
- тіло відповіді.

Приклади статусів:

- `200 OK` — запит успішно оброблено;
- `404 Not Found` — ресурс не знайдено;
- `500 Internal Server Error` — внутрішня помилка сервера.

Статус не описує сам формат документа. Для цього використовується окремий заголовок.

---

## 9. MIME-типи та заголовок Content-Type

Користувачі настільних операційних систем часто визначають тип файлу за його розширенням: `.html`, `.png`, `.pdf` тощо.

Для браузера адреса або розширення не є надійним джерелом інформації. Сервер повинен повідомити тип переданих даних у заголовку:

```http
Content-Type: text/html
```

Це значення називають MIME-типом або медіатипом.

Приклади:

```text
text/html
text/plain
text/css
application/javascript
application/json
image/png
image/jpeg
application/pdf
```

Залежно від `Content-Type` браузер вирішує, як обробити відповідь:

- HTML — розібрати як розмітку;
- CSS — застосувати як таблицю стилів;
- JavaScript — виконати як програмний код;
- зображення — відобразити;
- JSON — передати програмі або показати як структурований текст;
- PDF — відкрити у вбудованому переглядачі або запропонувати зберегти.

Неправильно вказаний тип може призвести до некоректного відображення або до проблем безпеки.

## 10. Developer Tools

У сучасних браузерах є інструменти розробника. Зазвичай їх можна відкрити клавішею `F12` або комбінацією `Ctrl+Shift+I`.

Основні вкладки:

- **Elements** — структура HTML та стилі;
- **Console** — повідомлення й помилки JavaScript;
- **Network** — мережеві запити;
- **Sources** — завантажені скрипти й налагодження;
- **Application** — локальне сховище, cookie та інші дані.

У вкладці Network можна побачити:

- які ресурси запросив браузер;
- адресу кожного запиту;
- HTTP-метод;
- статус відповіді;
- час виконання;
- заголовки запиту й відповіді;
- значення `Content-Type`;
- тіло відповіді.

Це один із головних інструментів веброзробника.

## 11. Будова URL

URL визначає адресу ресурсу. Загальний вигляд:

```text
scheme://host:port/path?query#fragment
```

Наприклад:

```text
https://example.com:443/catalog/books?page=2#results
```

### 11.1. Схема

```text
https
```

Схема визначає спосіб доступу до ресурсу. Для вебу найчастіше використовуються:

- `http`;
- `https`.

HTTPS — це захищений варіант HTTP, у якому дані передаються через криптографічно захищене з’єднання.

### 11.2. Host

```text
example.com
```

Це ім’я вузла. Замість доменного імені може бути вказана IP-адреса.

Комп’ютери в мережі адресуються за допомогою IP-адрес. У IPv4 адреса зазвичай записується як чотири числа, наприклад:

```text
192.0.2.10
```

Людині незручно запам’ятовувати числові адреси, тому використовуються доменні імена.

### 11.3. DNS

**DNS — Domain Name System** — система, яка перетворює доменні імена на IP-адреси.

Коли користувач вводить домен, операційна система або браузер звертається до DNS-сервера й отримує адресу комп’ютера, з яким потрібно встановити з’єднання.

Перетворення можна продемонструвати командами:

```bash
ping example.com
```

або:

```bash
nslookup example.com
```

### 11.4. Порт

```text
443
```

IP-адреса визначає комп’ютер, але на одному комп’ютері можуть одночасно працювати різні мережеві програми. Порт визначає конкретну службу.

Порт можна порівняти з номером квартири у будинку:

- IP-адреса — адреса будинку;
- порт — номер квартири.

Типові порти:

- HTTP — `80`;
- HTTPS — `443`.

Якщо стандартний порт не вказаний у URL, браузер підставляє його автоматично.

### 11.5. Шлях

```text
/catalog/books
```

Історично шлях часто відповідав шляху до файлу на диску сервера. Наприклад, сервер міг буквально відкрити певний HTML-файл.

У сучасних вебзастосунках шлях часто є віртуальним. Серверний маршрутизатор аналізує його і викликає відповідний обробник.

Наприклад:

```text
/users/15
```

не обов’язково означає файл `15`. Це може бути команда серверу знайти користувача з ідентифікатором 15 та сформувати відповідь.

### 11.6. Параметри запиту

Після символу `?` можуть передаватися параметри:

```text
?page=2&sort=price
```

Пари `ім’я=значення` розділяються символом `&`.

Параметри можуть визначати:

- сторінку результатів;
- порядок сортування;
- фільтр;
- пошуковий запит;
- режим відображення.

### 11.7. Фрагмент

Після символу `#` може бути вказаний фрагмент:

```text
#results
```

Він зазвичай указує на певну частину вже завантаженого документа. Фрагмент традиційно обробляється браузером і не надсилається серверу у звичайному HTTP-запиті.

## 12. Як браузер завантажує сторінку

Коли користувач відкриває адресу, браузер зазвичай отримує не один файл.

Спочатку він запитує основний HTML-документ. Після аналізу HTML браузер знаходить посилання на інші ресурси:

- CSS;
- JavaScript;
- зображення;
- шрифти;
- аудіо;
- відео;
- додаткові документи.

Після цього браузер надсилає окремі запити для кожного ресурсу.

Наприклад:

```html
<link rel="stylesheet" href="/css/style.css">
<script src="/js/app.js"></script>
<img src="/images/logo.png" alt="Логотип">
```

CSS також може посилатися на зовнішні ресурси:

```css
body {
    background-image: url("/images/background.png");
}
```

JavaScript може формувати адреси програмно та надсилати запити до API.

Таким чином, одна видима сторінка може бути результатом десятків або сотень мережевих запитів.

---

## 13. Абсолютні та відносні адреси

**Абсолютна адреса** містить повну інформацію:

```text
https://example.com/images/logo.png
```

**Відносна адреса** визначається відносно поточного документа або кореня сайту:

```text
images/logo.png
```

або:

```text
/images/logo.png
```

Відносні адреси зручні, коли ресурси розміщуються на тому самому сайті.

---

## 14. Статична і динамічна відповідь

Вебсервер може обробити URL різними способами.

### Статичний ресурс

Сервер знаходить готовий файл і передає його без змін:

```text
/images/logo.png
/css/style.css
```

### Динамічний ресурс

Сервер викликає програмний код:

1. аналізує параметри;
2. перевіряє користувача;
3. звертається до бази даних;
4. виконує обчислення;
5. формує HTML або JSON;
6. надсилає результат.

Зовні для клієнта обидва випадки виглядають як запит і відповідь. Клієнт не зобов’язаний знати, чи існувала відповідь у вигляді готового файлу.

## 15. API і різні типи клієнтів

Універсальність API можна показати на прикладі бібліотечної системи.

Сервер підтримує запит:

```text
GET /api/books
```

і повертає:

```json
[
  {
    "id": 1,
    "title": "Основи вебтехнологій"
  }
]
```

Цю саму відповідь можуть використати:

- браузерний застосунок;
- мобільна програма;
- Telegram-бот;
- адміністративна панель;
- інший сервер.

Кожен клієнт відображає дані у власний спосіб. Браузер може побудувати HTML-таблицю, мобільний застосунок — нативний список, а бот — текстове повідомлення.

---

## 16. Архітектурне ускладнення серверної частини

Невеликий застосунок може складатися з одного серверного процесу та однієї бази даних.

У великих системах серверна частина може містити багато компонентів:

- сервіс автентифікації;
- сервіс користувачів;
- сервіс платежів;
- сервіс повідомлень;
- файлове сховище;
- пошуковий сервіс;
- черги фонових завдань.

Такий підхід часто називають мікросервісною архітектурою. На першому етапі достатньо розуміти, що слово «сервер» не завжди означає одну програму або один фізичний комп’ютер.

## 17. Основні висновки

1. Вебтехнології використовуються для створення розподілених систем, компоненти яких взаємодіють через мережу.
2. Браузер є типовим, але не єдиним вебклієнтом.
3. Frontend відповідає переважно за інтерфейс, backend — за серверну логіку й дані.
4. HTML описує структуру, CSS — оформлення, JavaScript — поведінку.
5. Перші вебсторінки були статичними файлами.
6. Серверне програмування дало змогу генерувати відповіді динамічно.
7. AJAX дозволив отримувати нові дані без повного перезавантаження сторінки.
8. Сучасні сервери часто надають API, а інтерфейс будується клієнтським кодом.
9. HTTP визначає правила запитів і відповідей.
10. Тип відповіді передається в заголовку `Content-Type`.
11. URL містить схему, вузол, порт, шлях, параметри та фрагмент.
12. DNS перетворює доменне ім’я на IP-адресу.
13. Одна вебсторінка зазвичай складається з багатьох окремо завантажених ресурсів.
14. Developer Tools дають змогу безпосередньо досліджувати роботу вебзастосунку.

## 19. Практична демонстрація

Для закріплення матеріалу доцільно виконати коротку демонстрацію:

1. Відкрити будь-яку нескладну вебсторінку.
2. Запустити Developer Tools.
3. Перейти на вкладку Network.
4. Перезавантажити сторінку.
5. Знайти основний HTML-запит.
6. Переглянути:
   - Request URL;
   - Request Method;
   - Status Code;
   - Content-Type.
7. Знайти окремі запити до CSS, JavaScript і зображень.
8. Порівняти їхні MIME-типи.
9. Розібрати URL одного з ресурсів на складові.
10. Виконати `nslookup` для доменного імені та показати отриману IP-адресу.
