# Лекція 7. Знайомство з React і створення першого проєкту

## 1. Від статичної сторінки до динамічного інтерфейсу

HTML створювався передусім як засіб опису структури документів. Такий документ зазвичай має відносно стабільний вміст: заголовки, абзаци, таблиці, списки, посилання та зображення.

Сучасний вебзастосунок поводиться інакше. Його інтерфейс постійно змінюється:

- користувач відкриває і закриває панелі;
- змінює фільтри;
- додає товари до кошика;
- ставить оцінки;
- отримує нові дані із сервера;
- переходить між різними станами без повного перезавантаження сторінки.

У звичайному JavaScript розробник може знайти елемент DOM і змінити його вручну:

```javascript
const counter = document.querySelector("#counter");
counter.textContent = "1";
```

Для невеликого прикладу цього достатньо. У складному інтерфейсі виникає необхідність постійно синхронізувати дані програми з DOM:

- який елемент потрібно змінити;
- чи треба його створити або видалити;
- яке значення має бути відображене;
- які інші частини сторінки залежать від тих самих даних;
- як повторно використати однакову частину інтерфейсу.

React пропонує інший підхід: розробник описує, яким повинен бути інтерфейс для поточного стану, а бібліотека визначає, які зміни потрібно внести до реального DOM.

## 2. Що таке React

React — JavaScript-бібліотека для побудови користувацьких інтерфейсів.

Основні ідеї React:

- інтерфейс складається з компонентів;
- компонент описує окрему логічну частину інтерфейсу;
- компоненти можна вкладати й повторно використовувати;
- зовнішні дані передаються через `props`;
- внутрішні змінні дані зберігаються у стані;
- компонент повертає опис інтерфейсу у формі JSX;
- React узгоджує цей опис із реальним DOM.

React передусім відповідає за представлення. Маршрутизація, робота з API, глобальний стан та інші задачі можуть вимагати додаткових бібліотек.

## 3. Декларативний та імперативний підходи

Імперативний код описує послідовність конкретних дій:

```javascript
const message = document.querySelector(".message");
message.textContent = "Збережено";
message.classList.add("message--success");
message.hidden = false;
```

У декларативному підході описується бажаний результат:

```jsx
<Message type="success" visible={true}>
    Збережено
</Message>
```

React самостійно визначає, які операції з DOM потрібно виконати.

Декларативність не означає, що в програмі немає алгоритмів. Обробники подій і бізнес-логіка залишаються звичайним JavaScript-кодом. Декларативним є передусім опис представлення.

## 4. Node.js, npm і Vite

Сучасний React-проєкт використовує інструменти, які:

- створюють структуру проєкту;
- встановлюють залежності;
- перетворюють JSX;
- обробляють модулі;
- запускають локальний сервер;
- автоматично оновлюють сторінку;
- формують виробничу збірку.

Ці інструменти часто запускаються через Node.js.

Перевірка встановлення:

```bash
node --version
npm --version
```

`npm` є менеджером пакетів. Він встановлює бібліотеки й інструменти, потрібні проєкту.

Vite — сучасний інструмент створення, запуску та складання клієнтських застосунків.

Створення проєкту:

```bash
npm create vite@latest
```

Або з явним шаблоном:

```bash
npm create vite@latest first-react-app -- --template react
```

Після створення:

```bash
cd first-react-app
npm install
npm run dev
```

У терміналі з’являється локальна адреса, наприклад:

```text
http://localhost:5173/
```

Сервер розробки відстежує зміни у файлах і швидко оновлює застосунок у браузері.

## 5. Основні npm-команди

У `package.json` зазвичай визначено скрипти:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "lint": "eslint .",
    "preview": "vite preview"
  }
}
```

Запуск:

```bash
npm run dev
npm run build
npm run lint
npm run preview
```

- `dev` запускає сервер розробки;
- `build` створює оптимізовану виробничу збірку;
- `lint` перевіряє код;
- `preview` дозволяє локально переглянути готову збірку.

## 6. Структура React-проєкту

Типова структура:

```text
first-react-app/
├── node_modules/
├── public/
├── src/
│   ├── assets/
│   ├── App.css
│   ├── App.jsx
│   ├── index.css
│   └── main.jsx
├── .gitignore
├── eslint.config.js
├── index.html
├── package.json
├── package-lock.json
├── README.md
└── vite.config.js
```

### `package.json`

Містить:

- назву проєкту;
- скрипти;
- залежності;
- залежності для розробки.

### `package-lock.json`

Фіксує конкретні версії пакетів. Його зазвичай додають до репозиторію.

### `node_modules`

Містить установлені пакети. Каталог не додають до Git, оскільки його можна відновити командою:

```bash
npm install
```

### `public`

Містить статичні файли, які копіюються без обробки модульною системою.

### `src`

Містить вихідний код застосунку: компоненти, стилі, модулі та ресурси.

### `index.html`

Є початковою HTML-сторінкою. У ній міститься контейнер:

```html
<div id="root"></div>
```

У цей елемент React монтує весь інтерфейс.

## 7. Точка входу

Файл `main.jsx` може виглядати так:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App.jsx";
import "./index.css";

createRoot(document.getElementById("root")).render(
    <StrictMode>
        <App />
    </StrictMode>
);
```

Послідовність роботи:

1. браузер завантажує `index.html`;
2. підключається `main.jsx`;
3. JavaScript знаходить елемент `root`;
4. `createRoot` створює корінь React;
5. React рендерить компонент `App`;
6. `App` формує дерево вкладених компонентів.

Пакет `react` містить основні механізми бібліотеки, а `react-dom` забезпечує інтеграцію з DOM браузера.

## 8. JSX

JSX — розширення синтаксису JavaScript, яке дозволяє описувати інтерфейс за допомогою HTML-подібної розмітки:

```jsx
const element = <h1>Вітаємо!</h1>;
```

JSX не є звичайним HTML і не виконується браузером безпосередньо. Vite перетворює його на звичайний JavaScript.

### Вирази у фігурних дужках

```jsx
const name = "Олена";

return <h1>Вітаємо, {name}!</h1>;
```

У дужках можна використовувати вирази:

```jsx
<p>{2 + 2}</p>
<p>{user.name}</p>
<p>{formatDate(date)}</p>
```

Довільну інструкцію `if` або цикл `for` безпосередньо вставити не можна. Їх виконують до `return` або замінюють виразами.

### Відмінності від HTML

```jsx
<div className="card">
    <img src={imageUrl} alt="Обкладинка" />
</div>
```

Основні відмінності:

- використовується `className`;
- теги потрібно закривати;
- JavaScript-значення записуються у `{}`;
- обробники мають назви `onClick`, `onChange`;
- багато властивостей використовують camelCase.

### Один кореневий елемент

Компонент повинен повертати один кореневий вузол:

```jsx
return (
    <div>
        <h1>Заголовок</h1>
        <p>Текст</p>
    </div>
);
```

Коли зайвий контейнер не потрібний, використовують Fragment:

```jsx
return (
    <>
        <h1>Заголовок</h1>
        <p>Текст</p>
    </>
);
```

## 9. Функціональні компоненти

Найпростіший компонент:

```jsx
function Welcome() {
    return <h2>Вітаємо на курсі!</h2>;
}
```

Використання:

```jsx
function App() {
    return (
        <main>
            <Welcome />
        </main>
    );
}
```

Назва компонента починається з великої літери. JSX-теги з малої літери React сприймає як звичайні DOM-елементи.

Компонент є функцією, але в JSX використовується як власний тег.

## 10. Дерево компонентів

Інтерфейс можна розділити так:

```text
App
├── Header
├── VideoList
│   ├── VideoCard
│   ├── VideoCard
│   └── VideoCard
└── Footer
```

Компонент повинен представляти логічно завершену частину інтерфейсу. Необов’язково створювати окремий компонент для кожного HTML-тегу.

Переваги:

- повторне використання;
- локалізація логіки;
- спрощення тестування;
- зрозуміла структура;
- менша кількість дублювання.

## 11. Винесення компонента в окремий файл

`VideoCard.jsx`:

```jsx
import "./VideoCard.css";

export default function VideoCard() {
    return (
        <article className="video-card">
            <h2>Назва відео</h2>
        </article>
    );
}
```

`App.jsx`:

```jsx
import VideoCard from "./components/VideoCard/VideoCard.jsx";

export default function App() {
    return (
        <main>
            <VideoCard />
        </main>
    );
}
```

Пов’язані файли можна зберігати разом:

```text
components/
└── VideoCard/
    ├── VideoCard.jsx
    └── VideoCard.css
```

## 12. Імпорт та експорт

Стандартний експорт:

```jsx
export default function VideoCard() {
    // ...
}
```

Імпорт:

```jsx
import VideoCard from "./VideoCard.jsx";
```

Іменований експорт:

```jsx
export function VideoCard() {
    // ...
}
```

Імпорт:

```jsx
import { VideoCard } from "./VideoCard.jsx";
```

Якщо функцію не експортовано, інший файл не зможе її імпортувати.

## 13. Властивості компонента — `props`

Компонент без параметрів завжди відображає однаковий вміст. Для повторного використання йому передають дані:

```jsx
<VideoCard title="Вступ до React" views={120} />
<VideoCard title="Компоненти" views={95} />
```

Компонент отримує об’єкт `props`:

```jsx
function VideoCard(props) {
    return (
        <article>
            <h2>{props.title}</h2>
            <p>Переглядів: {props.views}</p>
        </article>
    );
}
```

З деструктуризацією:

```jsx
function VideoCard({ title, views }) {
    return (
        <article>
            <h2>{title}</h2>
            <p>Переглядів: {views}</p>
        </article>
    );
}
```

Через `props` можна передавати:

- рядки;
- числа;
- логічні значення;
- масиви;
- об’єкти;
- функції;
- React-елементи.

`props` слід розглядати як незмінні вхідні дані. Компонент не повинен переписувати їх.

## 14. Відображення масиву

Дані можна винести в окремий масив:

```javascript
const videos = [
    { id: 1, title: "HTML", views: 120 },
    { id: 2, title: "CSS", views: 95 },
    { id: 3, title: "JavaScript", views: 150 }
];
```

Відображення через `map`:

```jsx
function App() {
    return (
        <main>
            {videos.map(video => (
                <VideoCard
                    key={video.id}
                    title={video.title}
                    views={video.views}
                />
            ))}
        </main>
    );
}
```

`map` створює новий масив React-елементів. Це дозволяє формувати інтерфейс без ручного повторення однакового JSX.

## 15. Властивість `key`

Кожен елемент динамічного списку повинен мати стабільний унікальний ключ:

```jsx
key={video.id}
```

`key` допомагає React визначити:

- який елемент залишився тим самим;
- який було видалено;
- який додано;
- який переміщено.

Поганий ключ:

```jsx
key={Math.random()}
```

Він змінюється на кожному рендері, тому React вважає елементи новими.

Індекс масиву може бути прийнятним лише для повністю стабільного списку. Для звичайних даних краще використовувати постійний ідентифікатор.

`key` використовується самим React і не передається компоненту як звичайний `props`.

## 16. Virtual DOM

Під час рендерингу React створює власне представлення дерева інтерфейсу в пам’яті.

Коли змінюються дані:

1. компоненти виконуються повторно;
2. формується нове дерево React-елементів;
3. воно порівнюється з попереднім;
4. визначаються необхідні зміни;
5. вони застосовуються до реального DOM.

Цю модель часто називають Virtual DOM.

Практична перевага полягає не лише у швидкості, а й у тому, що розробник описує бажаний результат, а React централізовано керує оновленнями DOM.

## 17. Події в React

Обробник передається як JavaScript-функція:

```jsx
function VideoCard() {
    function handleClick() {
        console.log("Натиснуто");
    }

    return <button onClick={handleClick}>Подобається</button>;
}
```

Потрібно передати посилання:

```jsx
onClick={handleClick}
```

а не викликати функцію під час рендерингу:

```jsx
onClick={handleClick()}
```

Для передавання аргументу використовують обгортку:

```jsx
onClick={() => handleLike(videoId)}
```

Обробник отримує об’єкт події:

```jsx
function handleClick(event) {
    console.log(event.currentTarget);
}
```

## 18. Чому звичайна змінна не оновлює інтерфейс

Помилковий приклад:

```jsx
function LikeButton() {
    let likes = 0;

    function handleClick() {
        likes += 1;
        console.log(likes);
    }

    return (
        <button onClick={handleClick}>
            Подобається: {likes}
        </button>
    );
}
```

Значення в консолі змінюється, але інтерфейс не оновлюється.

Причини:

- `likes` є звичайною локальною змінною;
- React не отримує повідомлення про зміну;
- повторний рендеринг не планується;
- під час майбутнього виклику компонента змінна знову буде створена зі значенням `0`.

Дані, які впливають на інтерфейс і повинні зберігатися між рендерами, потрібно зберігати у стані.

## 19. Хук `useState`

```jsx
import { useState } from "react";

function LikeButton() {
    const [likes, setLikes] = useState(0);

    function handleClick() {
        setLikes(likes + 1);
    }

    return (
        <button onClick={handleClick}>
            Подобається: {likes}
        </button>
    );
}
```

`useState(0)` повертає два значення:

1. поточний стан;
2. функцію його оновлення.

Деструктуризація:

```javascript
const [likes, setLikes] = useState(0);
```

Setter повідомляє React, що компонент потрібно повторно відрендерити.

## 20. Незалежний стан компонентів

```jsx
<LikeButton />
<LikeButton />
<LikeButton />
```

Кожен екземпляр має власний стан. Натискання першої кнопки не змінює інші.

Одна функція компонента може створити багато незалежних екземплярів.

## 21. Функціональне оновлення стану

Замість:

```jsx
setLikes(likes + 1);
```

часто доцільно використовувати:

```jsx
setLikes(previousLikes => previousLikes + 1);
```

Це особливо важливо для кількох послідовних оновлень:

```jsx
setLikes(value => value + 1);
setLikes(value => value + 1);
```

React може групувати оновлення, а функціональна форма отримує актуальне попереднє значення.

## 22. Чому після setter-а видно старе значення

```jsx
function handleClick() {
    setLikes(likes + 1);
    console.log(likes);
}
```

У консолі буде старе значення.

Setter не переписує локальну константу `likes` у вже виконуваному обробнику. Він планує наступний рендеринг.

Послідовність:

```text
обробник бачить likes = 0
→ викликає setLikes(1)
→ React планує оновлення
→ обробник завершується
→ компонент виконується повторно
→ новий рендер отримує likes = 1
```

Стан можна уявляти як знімок для конкретного рендера.

## 23. Повторний рендеринг

Після зміни стану:

1. React повторно викликає функцію компонента;
2. компонент повертає новий JSX;
3. React порівнює його з попереднім результатом;
4. оновлюється потрібна частина DOM.

Повторний виклик функції компонента не означає повне перемальовування сторінки.

## 24. Правила хуків

Хуки потрібно викликати:

- лише у функціональних компонентах або власних хуках;
- на верхньому рівні функції;
- не всередині умов, циклів чи вкладених функцій.

Неправильно:

```jsx
if (enabled) {
    const [value, setValue] = useState(0);
}
```

React покладається на стабільний порядок виклику хуків між рендерами.

## 25. `StrictMode`

Точка входу часто містить:

```jsx
<StrictMode>
    <App />
</StrictMode>
```

У режимі розробки React може додатково викликати компоненти, щоб виявити:

- побічні ефекти під час рендерингу;
- неправильне очищення ресурсів;
- залежність від одноразового виконання.

Через це `console.log` у компоненті може з’являтися двічі.

Це механізм діагностики середовища розробки. У виробничій збірці такого додаткового виклику немає.

## 26. React Developer Tools

Розширення React Developer Tools додає вкладки:

- Components;
- Profiler.

У Components можна:

- переглянути дерево React-компонентів;
- вибрати компонент;
- побачити його `props`;
- дослідити стан;
- знайти вкладені компоненти.

Підсвічування оновлень показує, які компоненти повторно рендеряться.

Profiler допомагає аналізувати тривалість рендерингів і шукати зайві оновлення.

Вкладка Elements показує реальний DOM, а Components — логічне дерево React-компонентів.

## 27. Підсумковий приклад

`VideoCard.jsx`:

```jsx
import { useState } from "react";
import "./VideoCard.css";

export default function VideoCard({ title, image, views }) {
    const [likes, setLikes] = useState(0);

    function handleLike() {
        setLikes(value => value + 1);
    }

    return (
        <article className="video-card">
            <img
                className="video-card__image"
                src={image}
                alt=""
            />

            <div className="video-card__content">
                <h2>{title}</h2>
                <p>Переглядів: {views}</p>

                <button type="button" onClick={handleLike}>
                    Подобається: {likes}
                </button>
            </div>
        </article>
    );
}
```

`App.jsx`:

```jsx
import VideoCard from "./components/VideoCard/VideoCard.jsx";
import videos from "./data/videos.js";

export default function App() {
    return (
        <main className="video-list">
            {videos.map(video => (
                <VideoCard
                    key={video.id}
                    title={video.title}
                    image={video.image}
                    views={video.views}
                />
            ))}
        </main>
    );
}
```

У прикладі використано:

- функціональні компоненти;
- модулі;
- `props`;
- деструктуризацію;
- `map`;
- `key`;
- `onClick`;
- локальний стан `useState`;
- незалежний стан кожної картки.

## 28. Типові помилки

### Назва компонента з малої літери

Назва власного компонента повинна починатися з великої літери.

### Відсутній експорт

Компонент не можна імпортувати, якщо його не експортовано.

### Виклик обробника під час рендерингу

Неправильно:

```jsx
<button onClick={handleClick()}>
```

Правильно:

```jsx
<button onClick={handleClick}>
```

### Зміна `props`

Властивості є вхідними даними й не повинні переписуватися компонентом.

### Звичайна змінна замість стану

Зміна локальної змінної не викликає повторний рендеринг.

### Нестабільний `key`

Випадковий ключ руйнує ідентичність елементів між рендерами.

### Очікування негайної зміни стану

Setter планує новий рендер, але не змінює значення у поточному замиканні.

## 29. Підсумки

Основна модель React:

```text
дані
→ props і state
→ виконання компонентів
→ JSX-дерево
→ порівняння з попереднім представленням
→ необхідні зміни DOM
```

Компонент:

```text
отримує props
+ читає state
+ обробляє події
→ повертає опис інтерфейсу
```

Для створення першого застосунку потрібно:

1. встановити Node.js;
2. створити проєкт через Vite;
3. встановити залежності;
4. запустити сервер розробки;
5. розібрати точку входу;
6. створити власний компонент;
7. передати йому `props`;
8. сформувати список через `map`;
9. додати `key`;
10. керувати локальним станом через `useState`.

React дозволяє працювати не з окремими командами зміни DOM, а з компонентами та станами інтерфейсу. Це створює основу для подальшого вивчення ефектів, форм, HTTP-запитів, маршрутизації та керування спільним станом.
