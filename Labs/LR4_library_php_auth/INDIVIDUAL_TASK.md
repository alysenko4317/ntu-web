# ЛР4 — індивідуальне завдання (чернетка для методички)

> Текст нижче призначений для розділу **«Індивідуальне завдання»** у `lab4_php_auth.docx`.  
> Базовий механізм реєстрації, входу, сесій і захищеного кабінету описано в теоретичній частині та показано в еталонному коді репозиторію. **Самостійно** студент реалізовує два додаткові сценарії серверної обробки даних.

---

## Індивідуальне завдання

Після реалізації (на базі теоретичного матеріалу та/або еталону) базового механізму реєстрації, авторизації, HTTP-сесій і захищеного кабінету студент **самостійно** доповнює PHP-версію **власного** сайту двома функціональними блоками.

Завдання виконується на сайті обраної в ЛР1 тематики. У демо-еталоні репозиторію (бібліотека) ці блоки **навмисно не реалізовані повністю**: контактна форма залишена з клієнтською перевіркою, а кабінет показує дані лише для читання.

### Частина 1. Серверна обробка контактної форми

**Мета:** закріпити навички серверної обробки POST-запиту на прикладі **публічної** (не захищеної) сторінки.

На сторінці контактів (`contacts.php` або еквівалент у власному проєкті) потрібно:

1. Перевести форму зворотного зв’язку на метод `POST` і серверну обробку у тому ж файлі (за аналогією з `register.php`).
2. Виконати **серверну** перевірку полів (ім’я, email, текст повідомлення): обов’язковість, мінімальна довжина, формат email.
3. При помилках — показати повідомлення про помилки і **зберегти** введені користувачем значення у полях форми.
4. При успішній перевірці — зберегти повідомлення у файловому сховищі, наприклад `data/messages.json`, у вигляді масиву об’єктів із полями:
   - `name`
   - `email`
   - `message`
   - `created_at` (дата/час у форматі ISO 8601, наприклад через `date('c')`)
5. Після збереження застосувати схему **POST/Redirect/GET**: перенаправити користувача на ту саму сторінку з параметром успіху (наприклад `contacts.php?sent=1`) і показати повідомлення про успішне надсилання.
6. Забезпечити право запису для веб-сервера у каталог `data/` (аналогічно `users.json`).

**Рекомендації:**

- Винести читання/запис повідомлень у допоміжні функції (наприклад `includes/messages.php`).
- Клієнтську JavaScript-перевірку можна залишити як додаткову (з `novalidate` на формі або без неї), але **зарахування** — лише за наявності коректної серверної валідації.
- Якщо авторизований користувач відкриває форму, бажано **попередньо заповнити** поля імені та email з даних сесії/профілю (необов’язково, але заохочується).

**Приклад структури запису в `messages.json`:**

```json
[
  {
    "name": "Олена",
    "email": "olena@example.com",
    "message": "Підкажіть, будь ласка, режим роботи.",
    "created_at": "2026-06-29T14:30:00+03:00"
  }
]
```

### Частина 2. Редагування профілю в особистому кабінеті

**Мета:** закріпити роботу із захищеною сторінкою, файловим сховищем користувачів і оновленням даних поточного користувача.

На сторінці кабінету (`cabinet.php`) потрібно:

1. Залишити відображення поточних даних профілю (як мінімум: ім’я, email та одне **доменне** поле, обране студентом при реєстрації).
2. Додати форму **редагування профілю** (метод `POST`) з можливістю змінити:
   - **ім’я** користувача;
   - **одне доменне поле** (у демо-еталоні — «зал бібліотеки»; у власному проєкті — відповідне поле тематики);
   - поля **email** і унікального ідентифікатора (у демо — «читацький квиток») **не змінювати** через цю форму.
3. Виконати серверну перевірку нових значень (порожні поля, мінімальна довжина імені, допустимі значення для поля з вибором тощо).
4. Реалізувати функцію оновлення користувача у `includes/users.php` (наприклад `updateUserByEmail()`): знайти запис поточного користувача в `users.json`, оновити поля, зберегти файл.
5. Після успішного оновлення показати повідомлення про успіх (або перенаправити з параметром `?updated=1`).
6. Сторінка, як і раніше, має бути доступна **лише авторизованому** користувачу (`requireLogin()` на початку файлу).

**Рекомендації:**

- Оновлювати дані **тільки** користувача, чий email збігається з `$_SESSION['user_email']`.
- Не приймати email для оновлення з `$_POST` як довірений ідентифікатор — брати його лише з сесії.

## Критерії оцінювання індивідуальної частини

| Критерій | Частина 1 (контакти) | Частина 2 (кабінет) |
|----------|----------------------|---------------------|
| Форма надсилає POST на сервер | ✓ | ✓ |
| Серверна валідація обов’язкових полів | ✓ | ✓ |
| Збереження даних у JSON-файл | `messages.json` | оновлення `users.json` |
| POST/Redirect/GET або еквівалентний UX після успіху | ✓ | бажано |
| Захист сторінки кабінету | — | ✓ |
| Оновлення лише даних поточного користувача | — | ✓ |
| Допоміжні функції винесено в `includes/` | бажано | бажано |

## Що включити до звіту (додатково до базових вимог ЛР4)

- короткий опис двох самостійно реалізованих сценаріїв;
- фрагменти коду серверної валідації та збереження `messages.json`;
- фрагмент функції оновлення профілю в `users.json`;
- скріншоти: успішне надсилання контактної форми, помилка валідації, оновлений профіль у кабінеті.
