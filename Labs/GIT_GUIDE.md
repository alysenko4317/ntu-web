# Робота з Git — памʼятка для студента

Цей довідник допоможе організувати акуратну роботу з Git-репозиторієм упродовж усього курсу лабораторних робіт.

> Git — це не лише інструмент збереження коду, а й **щоденник вашого проєкту**. Чим якісніше ви ведете історію, тим простіше повернутися до будь-якої точки, зрозуміти, чому було прийнято те чи інше рішення, і працювати в команді.

---

## 1. Підготовка

### Створення акаунту

Якщо у вас ще немає акаунту — зареєструйтесь на одній із платформ:

| Платформа | URL | Примітка |
|-----------|-----|----------|
| **GitHub** | [github.com](https://github.com) | Найпопулярніша, безкоштовні приватні репозиторії |
| **GitLab** | [gitlab.com](https://gitlab.com) | Вбудований CI/CD, безкоштовні приватні репозиторії |

### Початкове налаштування Git

Після встановлення Git виконайте один раз:

```bash
git config --global user.name "Ваше Ім'я"
git config --global user.email "your.email@example.com"
```

Ці дані відображатимуться у кожному коміті.

### Створення репозиторію

```bash
# Створити новий локальний репозиторій
git init

# Або клонувати існуючий
git clone https://github.com/username/repo.git
```

Після створення репозиторію на GitHub/GitLab — підключіть віддалений сервер:

```bash
git remote add origin https://github.com/username/repo.git
```

## 2. Щоденна робота

### Базовий цикл

```bash
# 1. Перевірити стан
git status

# 2. Переглянути зміни
git diff

# 3. Додати файли до індексу
git add .                    # усі змінені файли
git add src/index.html       # конкретний файл

# 4. Зафіксувати коміт
git commit

# 5. Відправити на сервер
git push
```

### Що НЕ варто комітити

Створіть файл `.gitignore` у корені репозиторію:

```gitignore
# Системні файли
.DS_Store
Thumbs.db
desktop.ini

# IDE
.idea/
.vscode/
*.swp

# Залежності
node_modules/
vendor/

# Конфіденційні дані
.env
*.log
```

## 3. Правила оформлення комітів

Коміт — це **одиниця змін** із зрозумілим описом. Хороший коміт дозволяє будь-кому (і вам через місяць) зрозуміти, **що** було зроблено і **чому**.

### Структура повідомлення

```
<тип>: <короткий опис змін> (до ~72 символів)

<пояснення: чому зроблено, які рішення прийнято>
<що саме змінилось, якщо не очевидно з коду>
```

**Перший рядок** — заголовок коміту. Він має бути:
- лаконічним (до 72 символів);
- дієслово у наказовому способі: «додати», «виправити», «оновити» (не «додав», «додано»).

**Тіло** (після порожнього рядка) — деталі. Пишіть, коли зміна нетривіальна.

### Типи комітів

| Тип | Коли використовувати | Приклад |
|-----|----------------------|---------|
| `feat` | Нова функціональність | `feat: додати сторінку каталогу з картками книг` |
| `fix` | Виправлення помилки | `fix: виправити валідацію email у формі реєстрації` |
| `refactor` | Рефакторинг без зміни поведінки | `refactor: винести навігацію у спільний layout` |
| `style` | Зміни CSS, форматування | `style: оновити кольори кнопок відповідно до макету` |
| `docs` | Документація | `docs: додати інструкцію з запуску у README` |
| `chore` | Конфігурація, інфраструктура | `chore: додати Dockerfile для Apache` |

### Приклади

**Погано:**

```
fix
```
```
update
```
```
зміни
```
```
ааа нарешті запрацювало
```

**Добре:**

```
feat: додати форму реєстрації з валідацією полів

Форма містить поля: ім'я, email, пароль, підтвердження пароля.
Клієнтська перевірка: обов'язковість полів, формат email,
мінімальна довжина пароля (6 символів), збіг паролів.
```

```
fix: виправити помилку 404 при переході на /catalog

Причина: у routes.php був зайвий слеш у шаблоні URI.
Маршрут "//catalog" замінено на "catalog".
```

```
refactor: перенести логіку авторизації з login.php у AuthService

Контролер тепер лише приймає запит і викликає сервіс.
Бізнес-логіка (перевірка пароля, створення сесії) — в AuthService.
Це спрощує подальше тестування та повторне використання.
```

### Як писати тіло коміту зручно

За замовчуванням `git commit` (без `-m`) відкриває текстовий редактор, де можна написати багаторядкове повідомлення. Якщо це незручно, можна передати кілька рядків через повторення прапорця `-m`:

```bash
git commit -m "feat: додати сторінку контактів" -m "Форма з полями ім'я, email, повідомлення. Поки без серверної обробки — лише HTML-розмітка та CSS."
```

## 4. Робота з гілками

Гілки дозволяють працювати над новою функцією ізольовано від основного коду.

### Модель гілкування для лабораторних

```
main (або master)
 │
 ├── lr1-static-site        ← ЛР1: статичний сайт
 │     └── (merge → main)
 │
 ├── lr3-javascript          ← ЛР3: додати JavaScript
 │     └── (merge → main)
 │
 ├── lr4-php-auth            ← ЛР4: PHP авторизація
 │     └── (merge → main)
 │
 └── ...
```

### Основні команди

```bash
# Створити нову гілку і перейти на неї
git checkout -b lr4-php-auth

# Перейти на існуючу гілку
git checkout main

# Переглянути список гілок
git branch

# Переглянути всі гілки (включно з віддаленими)
git branch -a

# Видалити гілку (після злиття)
git branch -d lr4-php-auth
```

### Правила іменування гілок

| Шаблон | Приклад | Коли |
|--------|---------|------|
| `lr<N>-<опис>` | `lr4-php-auth` | Робота над конкретною ЛР |
| `feature/<опис>` | `feature/contact-form` | Окрема фіча в межах ЛР |
| `fix/<опис>` | `fix/login-redirect` | Виправлення помилки |

Імена гілок — **латиницею**, **малими літерами**, слова через дефіс.

## 5. Злиття гілок (Merge)

Коли робота в гілці завершена, її потрібно злити в `main`.

### Звичайне злиття (merge)

```bash
git checkout main
git merge lr4-php-auth
```

Зберігає **всю історію комітів** з гілки. Використовуйте, коли:

- коміти в гілці акуратні та осмислені;
- кожен коміт — логічна одиниця роботи;
- ви хочете зберегти детальну хронологію.

### Squash merge (злиття зі зжиманням)

```bash
git checkout main
git merge --squash lr4-php-auth
git commit -m "feat: додати PHP-авторизацію (ЛР4)"
```

Зжимає **всі коміти** гілки в один. Використовуйте, коли:

- у гілці багато дрібних, хаотичних комітів (`fix`, `fix2`, `oops`, `test`);
- історія гілки не несе цінності;
- ви хочете, щоб в `main` з'явився **один чистий коміт** із підсумком роботи.

### Коли що обирати?

| Ситуація | Рекомендація |
|----------|-------------|
| Коміти чисті, кожен — змістовна одиниця | `git merge` (звичайне злиття) |
| Коміти хаотичні, є «WIP», «fix typo», «test» | `git merge --squash` |
| Feature-гілка з 1–3 комітами | `git merge` |
| Feature-гілка з 20+ мілких комітів | `git merge --squash` |

> **Порада:** якщо ви одразу пишете якісні коміти, squash зазвичай не потрібен. Краще привчити себе до акуратної історії, ніж постійно зжимати хаос.

### Конфлікти злиття

Якщо Git не може автоматично злити зміни, він повідомить про конфлікт:

```bash
git merge lr4-php-auth
# CONFLICT (content): Merge conflict in src/index.php
```

Що робити:

1. Відкрити файл із конфліктом — знайти маркери `<<<<<<<`, `=======`, `>>>>>>>`
2. Вирішити конфлікт: залишити потрібну версію або поєднати обидві
3. Видалити маркери конфлікту
4. Зберегти файл
5. `git add <файл>` і `git commit`

GUI-клієнти (див. нижче) значно спрощують вирішення конфліктів візуально.

## 6. GUI-клієнти для Git

Командний рядок — базовий і універсальний інструмент, але графічні клієнти роблять щоденну роботу зручнішою: візуалізація історії, diff-перегляд, вирішення конфліктів.

### Рекомендовані клієнти

| Клієнт | Платформа | Ціна | Особливості |
|--------|-----------|------|-------------|
| [**TortoiseGit**](https://tortoisegit.org/) | Windows | Безкоштовний | Інтеграція в Провідник Windows (контекстне меню). Зручний для тих, хто звик працювати у файловому менеджері. Показує статус файлів іконками-оверлеями. |
| [**Sourcetree**](https://www.sourcetreeapp.com/) | Windows, macOS | Безкоштовний | Потужна візуалізація гілок та історії. Підтримує Git Flow. Добре підходить для macOS. |
| [**git cola**](https://git-cola.github.io/) | Linux, WSL | Безкоштовний | Легкий GUI для Linux/WSL. Зручний staging area, diff-перегляд. |
| [**GitHub Desktop**](https://desktop.github.com/) | Windows, macOS | Безкоштовний | Простий інтерфейс, тісна інтеграція з GitHub. Підходить для початківців. |
| [**GitKraken**](https://www.gitkraken.com/) | Windows, macOS, Linux | Freemium | Сучасний інтерфейс, drag-and-drop для merge/rebase. Безкоштовний для публічних репозиторіїв. |

### Встановлення TortoiseGit (Windows)

1. Встановити [Git for Windows](https://git-scm.com/download/win)
2. Завантажити та встановити [TortoiseGit](https://tortoisegit.org/download/)
3. Після встановлення — клікнути правою кнопкою на папку проєкту в Провіднику → побачити меню TortoiseGit

### Встановлення git cola (WSL/Linux)

```bash
sudo apt update
sudo apt install git-cola
```

Запуск:

```bash
cd ~/my-project
git cola
```

> **Порада:** навіть якщо ви користуєтесь GUI-клієнтом, корисно розуміти, які команди він виконує «під капотом». Це допоможе у складних ситуаціях.

## 7. Рекомендований Git-воркфлоу для курсу

### На початку курсу (ЛР1)

```bash
# 1. Створити репозиторій на GitHub
# 2. Клонувати на свій комп'ютер
git clone https://github.com/username/web-course.git
cd web-course

# 3. Створити гілку для ЛР1
git checkout -b lr1-static-site

# 4. Працювати, комітити осмислені порції змін
git add .
git commit -m "feat: створити структуру HTML-сторінок"

git add .
git commit -m "style: додати CSS-стилі для навігації та hero-секції"

git add .
git commit -m "feat: додати сторінку каталогу з картками"

# 5. Відправити гілку на сервер
git push -u origin lr1-static-site

# 6. Злити в main (після завершення ЛР)
git checkout main
git merge lr1-static-site
git push
```

### Для кожної наступної ЛР

```bash
git checkout main
git pull
git checkout -b lr<N>-<короткий-опис>

# ... робота, коміти ...

git push -u origin lr<N>-<короткий-опис>

# Після завершення — злити в main
git checkout main
git merge lr<N>-<короткий-опис>    # або --squash якщо коміти хаотичні
git push
```

## 8. Типові помилки та як їх уникнути

| Помилка | Чому погано | Як правильно |
|---------|-------------|--------------|
| Один коміт на всю ЛР | Неможливо зрозуміти етапи роботи | Комітити логічні порції: сторінка, компонент, фіча |
| Коміти типу `fix`, `update`, `123` | Незрозуміло, що змінено | Описувати суть: `fix: виправити редирект після логіну` |
| Комітити `node_modules/`, `.env` | Засмічення репозиторію, витік секретів | Налаштувати `.gitignore` перед початком роботи |
| Працювати лише в `main` | Немає ізоляції, складно відкотити | Створювати гілку для кожної ЛР або фічі |
| Ніколи не пушити | Ризик втрати роботи | Пушити після кожного робочого сеансу |
| `git add .` без `git status` | Можна додати зайві файли | Завжди перевіряти `git status` або `git diff` перед комітом |

## 9. Корисні команди

```bash
# Переглянути історію (компактно)
git log --oneline --graph

# Переглянути зміни у конкретному коміті
git show abc1234

# Скасувати зміни у файлі (до коміту)
git checkout -- src/index.html

# Прибрати файл з індексу (unstage)
git reset HEAD src/index.html

# Переглянути різницю між гілками
git diff main..lr4-php-auth

# Переглянути, які файли змінено між гілками
git diff main..lr4-php-auth --stat

# Тимчасово зберегти незакомічені зміни
git stash
git stash pop
```
